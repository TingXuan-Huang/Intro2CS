0 · The goal

Code that works once vs code you can trust twice

Works once

"It printed the right number today."

One tangled cell; results printed and scrolled away; nothing named, nothing returned, nothing checkable.

Trust twice

Reusable next month, readable by a colleague, checkable automatically.

Named contracts that return values; a module you import; tests that prove the answers didn't move.

Correct today and trustworthy tomorrow are different properties — and only one of them survives contact with new data.

1 · Functions as contracts

A function is a promise with three visible parts

Signature

The name and the inputs: what you must supply.

revenue(quantity, unit_price)

Docstring

The promise: what it does and what it returns.

"""Revenue for one order line…"""

Return value

The deliverable: the answer handed back to the caller.

return quantity * unit_price

Read a function's first two lines and you should know how to use it without reading its body. That is the whole idea.
A function is not a way to avoid retyping. It is a contract other code can rely on.
1 · The contract in code

The caller reads the signature and docstring, not the body

def revenue(quantity: int, unit_price: float) -> float:
    """Revenue for one order line: quantity times unit price."""
    return quantity * unit_price

print("revenue(3, 2.5) =", revenue(3, 2.5))
print("the promise    :", revenue.__doc__)
revenue(3, 2.5) = 7.5
the promise    : Revenue for one order line: quantity times unit price.

Read it aloud

"Give me a whole number and a decimal; I hand back a decimal — the line's revenue."

The docstring lives on the function itself: revenue.__doc__. Help systems, editors, and colleagues all read it there.

Signature · docstring · return value — the contract. Everything else is implementation.
1 · Return, don't print

A printed answer is thrown away; a returned answer composes

Prints — and forgets

def print_revenue(quantity, unit_price):
    print(quantity * unit_price)

shown = print_revenue(3, 2.5)   # prints 7.5…
print(shown)                    # …but gave back:
7.5
None

Returns — and composes

def return_revenue(quantity, unit_price):
    return quantity * unit_price

handed = return_revenue(3, 2.5)
print(handed * 2)               # reusable!
15.0
Only a returned value can be stored, combined with others, and — in Unit 5 — tested. The caller of a printing function gets None.
1 · The strongest promise

Pure functions: same input, same output, nothing touched

Pure

Depends only on its inputs; changes nothing outside itself.

Call it a thousand times: the same answer, no surprises — and, in Unit 5, trivially easy to test.

Side-effecting

Reaches outside: prints, writes a file, or modifies its argument.

Not evil — writing your cleaned CSV to disk is the point of some functions — but effects belong at the edges of a program, clearly labelled.

The dangerous side effect is the quiet one: a function that silently edits the object its caller still holds.
1 · Mutation, caught in the act

The mutating version edits the caller's frame — Lesson 1's aliasing, again

def drop_cheap_MUTATING(orders):
    orders.drop(orders[orders["price"] < 1.0].index,
                inplace=True)   # edits in place
    return orders

def drop_cheap_PURE(orders):
    return orders[orders["price"] >= 1.0].reset_index(drop=True)

toy = pd.DataFrame({"item": ["a","b","c"], "price": [0.5, 2.0, 3.0]})
drop_cheap_PURE(toy)      # toy still has 3 rows — safe
drop_cheap_MUTATING(toy)  # toy now has 2 rows — surprise!
after PURE call,     original still has 3 rows
after MUTATING call, original now has   2 rows

Why it hurts

Every step downstream now works on a corrupted frame — and the crash, if any, happens far from this line.

Two names for one object (Lesson 1's aliasing) plus inplace=True is how "cleaning" becomes corruption.
1 · Purity is in your exercise contract

The checker demands purity, in writing

clean_prices(transactions) — "Return a NEW frame … Must not modify the frame passed in."

Why the contract insists

Cleaning runs early. If it mutated its input, every later step — country totals, the customer ranking — would silently read corrupted data.

How it will be enforced

Unit 5 shows the test: call clean_prices, then assert the input still has all its rows. Purity is a checkable property.

Keep cleaning functions pure; push printing and saving to the edges of the program.
1 · Type hints

Type hints are property declarations in the signature

def revenue(quantity: int, unit_price: float) -> float: …

def keep_real_prices(orders: pd.DataFrame) -> pd.DataFrame: …

# "-> pd.DataFrame" promises the caller a table, not a number.

Three facts about hints

  • Python does not enforce them at run time.
  • Humans read them as documentation.
  • Tools — editors, type checkers — verify them.
Lesson 1's central idea — a type is a claim about which operations are legitimate — written directly into the contract.
1 · Before

One long procedural paragraph — it works, and it grows unmanageable

# BEFORE — load, clean, summarise, all tangled together
raw = pd.read_csv("../course_data/online_retail_sample.csv")
raw["Description"] = raw["Description"].str.strip()
raw["revenue"] = raw["Quantity"] * raw["UnitPrice"]
raw = raw[raw["UnitPrice"] >= 1.0].reset_index(drop=True)
print("rows kept:", len(raw))
print("total revenue:", round(raw["revenue"].sum(), 2))

What's wrong — nothing, yet

It is correct. But no step has a name, nothing is returned, and the only output is print.

This is exactly the Lesson-2 cell shape that becomes a 300-line script under deadline.

The problem with tangled code is not that it is wrong. It is that you cannot reuse, verify, or safely change any part of it.
1 · After

Three named contracts — same numbers, new properties

def load_orders(path: str) -> pd.DataFrame:
    """Read the retail CSV, tidy Description, add revenue."""
    orders = pd.read_csv(path)
    orders["Description"] = orders["Description"].str.strip()
    orders["revenue"] = orders["Quantity"] * orders["UnitPrice"]
    return orders

def keep_real_prices(orders: pd.DataFrame) -> pd.DataFrame:
    """Return a NEW frame with rows priced at 1.00 or more (pure)."""
    return orders[orders["UnitPrice"] >= 1.0].reset_index(drop=True)

def total_revenue(orders: pd.DataFrame) -> float:
    """Grand total of revenue — a Lesson-4 scan — rounded to 2 dp."""
    return round(orders["revenue"].sum(), 2)

What changed

  • Each step has a name you can read.
  • Each has a docstring you can trust.
  • Each returns a value you can test.
  • keep_real_prices is pure by design.

The final exercise asks you to carve up a much uglier script exactly this way.

Refactoring did not change one number. It changed who can trust the code, and how.
1 · Checkpoint

Read a contract, predict the behavior

def clean_prices(transactions: pd.DataFrame) -> pd.DataFrame: — from the signature alone: what do you hand it, what comes back, and what does the word "pure" add that the hint cannot?
You hand it a DataFrame; you get a DataFrame back — the hints declare those properties, though Python won't enforce them. "Pure" adds the promise hints cannot express: the frame you passed in is unchanged afterwards, and the result is a new object. That promise is exactly what a test can check.
Contracts are read in layers: name, hints, docstring — then, in Unit 5, tests that make the promises enforceable.
2 · Functional tools

Small tools that replace loops: declare the rule once

Lesson 4 said

Most analytics is scanning, filtering, and combining values.

This unit says

Python has a compact, readable vocabulary for exactly those moves.

The punchline

A pandas chain is this idea — a pipeline of pure transformations.

Each tool takes a rule — an expression or a function — and applies it everywhere. You say what; Python runs the loop.
You have used pieces of this toolkit already. Today we name them deliberately.
2 · Comprehensions

One line builds a list, a dict, or a set from a rule

prices = [0.5, 2.0, 3.5, 0.8, 5.0]

# list: keep prices >= 1, express in pence
pence = [round(p * 100) for p in prices if p >= 1.0]

# dict: build a lookup (Lesson 4's O(1) friend)
price_label = {p: ("cheap" if p < 1.0 else "ok") for p in prices}

# set: unique first letters (uniqueness is a Lesson 1 property)
words = ["metal", "mug", "doormat", "mug"]
firsts = {w[0] for w in words}
list  : [200, 350, 500]
dict  : {0.5: 'cheap', 2.0: 'ok', 3.5: 'ok',
         0.8: 'cheap', 5.0: 'ok'}
set   : {'m', 'd'}

The brackets tell you the structure you get back: […] list, {k: v} dict, {…} set — each with its Lesson 1 properties.

Transform and filter in one readable line — it says what, not how.
2 · Comprehension anatomy

Three slots: expression, source, condition

[ round(p * 100) for p in prices if p >= 1.0 ]

Expression

round(p * 100) — the transform applied to each survivor.

Source

for p in prices — the scan (Lesson 4's Θ(n) walk).

Condition

if p >= 1.0 — the filter; rows failing it never reach the expression.

When the rule outgrows one readable line — nested conditions, multiple steps — fall back to a plain loop. Readability is the contract.
2 · sorted with key=

The key function defines what "comes first" means

customers = [
    {"id": 111, "total": 50.0},
    {"id": 999, "total": 50.0},
    {"id": 555, "total": 50.0},
    {"id": 222, "total": 90.0},
]

ranked = sorted(customers,
                key=lambda c: (-c["total"], c["id"]))

Read the lambda aloud

"Given one customer, return the value to compare them by."

A lambda is just a one-line unnamed function (Lesson 4). When the logic outgrows a line, promote it to a named def.

Lesson 4 built leaderboards this way. Today's addition is the tuple key — sorting by several things at once.
2 · The tuple-key trick

(-total, id): descending on one field, ascending on the tie-break

How tuples compare

Python compares the first fields; only on a tie does it look at the second.

Negating a numeric field flips its direction: biggest total becomes smallest key.

(-90.0, 222) < (-50.0, 111) < (-50.0, 555) < (-50.0, 999)

RankCustomerTotalWhy here
122290.0highest total
211150.0tie → smallest id
355550.0tie → next id
499950.0tie → largest id
"Sort by total descending, ties by id ascending" — precisely how the exercise's top_customers must break ties.

2 · map and filter

map + filter and a comprehension say the same thing

Nested calls, read inside-out
nums = [1, 2, 3, 4, 5, 6]

doubled_evens = list(
    map(lambda n: n * 2,
        filter(lambda n: n % 2 == 0, nums)))
One line, read left-to-right
doubled_evens = [n * 2 for n in nums
                        if n % 2 == 0]
map + filter : [4, 8, 12]
comprehension: [4, 8, 12]
same result  : True
Both are the functional idea — apply a rule everywhere. In Python, the comprehension is usually the more readable spelling; recognize map/filter when you meet them in other people's code.
2 · zip and enumerate

Walk two lists in step; count as you go

ids = [111, 222, 555]
totals = [50.0, 90.0, 50.0]

# zip pairs them up — and a dict comprehension indexes them
lookup = {cid: total for cid, total in zip(ids, totals)}

# enumerate hands you the running position — no manual counter
for rank, c in enumerate(ranked, start=1):
    print(f"#{rank}: customer {c['id']} spent {c['total']}")
zipped lookup: {111: 50.0, 222: 90.0, 555: 50.0}

2 · The bridge

A pandas chain is a functional pipeline

Step by step — a variable per step
step1 = orders[orders["Country"] == "United Kingdom"]
step2 = step1.groupby("StockCode")["revenue"].sum()
step3 = step2.sort_values(ascending=False)
top_uk = step3.head(3)
The same pipeline as one chain
top_uk = (
    orders
    .query("Country == 'United Kingdom'")   # filter
    .groupby("StockCode")["revenue"].sum()  # group
    .sort_values(ascending=False)           # order
    .head(3)                                # slice
)
Each verb takes a frame and returns a new one — Lesson 2's rule — so the steps snap together like pipe segments. The two spellings produce equal results.
2 · Chain discipline

Chains are for reading, not for showing off

Do

Wrap the chain in parentheses and put one verb per line — the pipeline reads top to bottom: filter, group, reduce, order, slice.

Pause

If a step needs a comment, or you want to inspect its result, give it a name. A variable is a breakpoint for humans.

Don't

A ten-method run-on that nobody can read is not functional style — it is the tangled cell again, rotated 90°.

Reading a chain aloud is reading the algorithm. When you can't read it aloud, break it.
2 · .apply vs vectorization

Same answer; a Python loop vs compiled column math

big = pd.concat([orders] * 300, ignore_index=True)  # ~15k rows

apply_rev = big.apply(
    lambda row: row["Quantity"] * row["UnitPrice"],
    axis=1)                       # a Python loop per row

vector_rev = big["Quantity"] * big["UnitPrice"]
                                  # one compiled operation
rows: 15300
  .apply row-wise :   0.19 s
  vectorized      :   0.0004 s
  same answer     : True
  speedup         : ~500x

Numbers vary by machine; the shape of the gap does not — Lesson 4's algorithm-vs-implementation distinction, live.

.apply is defensible when the per-row work truly can't be vectorized. It should be a choice, not a habit.
2 · Checkpoint

Read the key, predict the order

sorted(rows, key=lambda r: (r["country"], -r["revenue"])) — in plain English, what order does this produce?
Countries in ascending (alphabetical) order; within each country, revenue in descending order — the negation flips only the second field. Tuple position sets priority; sign sets direction.
If you can translate any tuple key into a sentence, you can write any leaderboard.

3 · The specimen

messy_analysis.py: correct, and everything this unit warns about

What it does

Loads the retail CSV, cleans prices, totals revenue for three countries, ranks the top 3 customers — and prints correct numbers for all of it.

House rule

3 · Exhibit B

The top-3 selection loop: three ifs deep

for rank in range(3):
    best_pos = -1
    for j in range(len(customer_ids)):
        if totals[j] is None:
            continue
        if best_pos == -1:
            best_pos = j
        elif totals[j] > totals[best_pos]:
            best_pos = j
        elif totals[j] == totals[best_pos] and \
             customer_ids[j] < customer_ids[best_pos]:
            best_pos = j
    print("  #" + str(rank + 1), "customer", …)
    totals[best_pos] = None   # "remove" by overwriting

Buried treasure

Inside the nesting hides a real business rule: ties go to the smaller customer id.

It works — Lesson 4's selection-scan pattern — but the rule is invisible unless you trace all three branches.

Compare: sorted(…, key=lambda …: (-total, id))[:3] states the same rule in one line.

Deep nesting doesn't make code wrong. It makes the rules inside it unreadable — and unreadable rules drift.
3 · The smell taxonomy

Four code smells, caught red-handed

Copy-paste blocks

Three near-identical country loops.

Change once → fix three places → miss one.

Magic numbers

>= 1.0, unexplained.

A price floor? A currency assumption? The code won't say.

Deep nesting

The top-3 loop, three ifs deep.

Hard to read; easy to get the tie-break subtly wrong.

Print, not return

Every result printed, nothing returned.

Nothing reusable, nothing testable (Unit 1).

None of these is a bug — the script is correct. They are what makes correct code impossible to trust, extend, or test.
3 · The cure

Extract functions — into a module you import

# retail_utils.py — your first multi-file project
"""Retail helpers, extracted from messy_analysis.py."""
import pandas as pd

PRICE_FLOOR = 1.0   # the magic 1.0, named once

def load_orders(path):
    """Read the CSV, tidy Description, add revenue."""
    orders = pd.read_csv(path)
    orders["Description"] = orders["Description"].str.strip()
    orders["revenue"] = orders["Quantity"] * orders["UnitPrice"]
    return orders

def keep_real_prices(orders):
    """Return a NEW frame priced at or above PRICE_FLOOR (pure)."""
    return orders[orders["UnitPrice"] >= PRICE_FLOOR].reset_index(drop=True)

Each smell, answered

  • Magic number → named constant PRICE_FLOOR.
  • Tangle → one job per function, docstring each.
  • Print → every function returns.
  • A file of helpers → a module any script can import.
A module is just a .py file of related contracts. %%writefile in the notebook writes it to disk instead of running it.

3 · Checkpoint

What does the import promise?

from retail_utils import revenue_by_country — name everything this single line asserts, and what happens when each assertion fails.
It asserts (1) a module named retail_utils can be found on sys.path — otherwise ModuleNotFoundError; and (2) that module defines a name revenue_by_country — otherwise ImportError. Both fail immediately, at the door, naming the missing piece — which is exactly how you want contracts to fail.

4 · Errors and defensive code

Refactored code still meets bad input — fail well

1 · Read

Tracebacks bottom-up: the last line is the fact, the frames above are the route.

2 · Catch

Only the narrow exception you expected — never a blanket.

3 · Raise

The instant a contract is broken — at the door, with a message that names the problem.

Failing well is a feature. A clear early failure is worth more than a mysterious late one — and infinitely more than a silent wrong answer.
4 · Read tracebacks bottom-up

Bottom = what went wrong; up = where it happened

def parse_price(text):
    return float(text)          # blows up on bad text

def load_prices(values):
    return [parse_price(v) for v in values]

load_prices(["1.95", "not-available", "2.10"])
Traceback (most recent call last):
  File "cell", line 7, in <module>
    load_prices(["1.95", "not-available", "2.10"])
  File "cell", line 5, in load_prices
    return [parse_price(v) for v in values]
  File "cell", line 2, in parse_price
    return float(text)
ValueError: could not convert string to float: 'not-available'

Reading order

Start at the bottom: the ValueError line is the fact — it even shows the offending text.

Then climb: parse_price raised it; load_prices called that; the cell called that.

The top line is where Python started, not where it broke.

A traceback is a map of the call stack at the moment of failure — read it bottom-up, always.

4 · try / except

Catch the narrow case you expected — and improve the message

def read_table(path):
    """Load a CSV; turn a missing file into an actionable message."""
    try:
        return pd.read_csv(path)
    except FileNotFoundError:
        raise FileNotFoundError(
            f"No file at {path!r} — check the path and your "
            "working directory (Lesson 2's first aid: "
            "print(Path.cwd()))."
        )

Only FileNotFoundError

A different failure — a permissions error, a malformed file — is not swallowed. It surfaces, which is exactly what you want.

The re-raise keeps the error type but upgrades the message with what the reader should do next.

try/except is for the narrow, expected case. It is a tool for better messages, not a lid for problems.
4 · The anti-pattern

A bare except is a lid on a boiling pot

Never this

try:
    result = analyse(df)
except:          # catches EVERYTHING
    result = None

This swallows the typo in analyse, the missing column, the keyboard interrupt — every future bug, silently.

Ask before you catch

  • Which specific exception do I expect here?
  • Can I actually do something about it?
  • If not both — don't catch. Let it surface.
Code that cannot fail loudly will eventually fail silently — and a silent wrong answer is the most expensive bug there is.
4 · Raise early

Validate at the door: raise the instant a contract is broken

REQUIRED = {"Quantity", "UnitPrice", "Country", "source_customer_id"}

def load_validated(path):
    """Load the retail CSV and FAIL LOUDLY if it isn't shaped as promised."""
    orders = pd.read_csv(path)
    missing = REQUIRED - set(orders.columns)
    if missing:
        raise ValueError(f"{path}: missing required columns {sorted(missing)}")
    if len(orders) == 0:
        raise ValueError(f"{path}: file has a header but zero rows")
    return orders

The set difference is the interview

REQUIRED − columns names exactly what's absent — a Lesson 1 set operation doing Lesson 2's "interview the data," in code.

Validate early — at the door, not in the kitchen. The check costs three lines; the debugging session it replaces costs an afternoon.
4 · The payoff

At the door vs five cells later

With the check

caught early: broken.csv: missing

5 · Testing with pytest

The payoff: returned values can be checked automatically

Unit 1 built it

Your functions return values instead of printing them.

So Unit 5 can

Compare those values against answers you verified by hand.

Lesson 3 runs it

pytest is a terminal tool — the command line work pays off here.

A test is a tiny experiment: set up known inputs, run the function, assert the answer. Do it once, and the computer re-runs the experiment forever.
This is where "trust twice" stops being a slogan and becomes a command you can run.
5 · The atom

assert — passes silently, fails loudly

def discounted(price, fraction):
    """Price after a fractional discount, rounded to 2 dp."""
    return round(price * (1 - fraction), 2)

assert discounted(100.0, 0.10) == 90.0   # passes silently
assert discounted(50.0, 0.0) == 50.0     # no discount, no change

assert discounted(100.0, 0.10) == 91.0, "expected 90.0"
AssertionError: expected 90.0

An assert is a claim: "this expression is True."

True → nothing happens. False → AssertionError, with your message attached.

Every test you will ever write is asserts arranged politely. Master this one keyword and pytest is just an organizer.
5 · The rhythm

Arrange, act, assert — every test, the same three beats

def test_discounted_ten_percent():
    price, fraction = 100.0, 0.10         # arrange
    result = discounted(price, fraction)  # act
    assert result == 90.0                 # assert
ArrangeBuild known inputs — small enough to verify by hand.
ActCall the function under test, once.
AssertState the answer you computed yourself.
Wrap each check in a function named test_… — that prefix is how pytest discovers what to run.
5 · A module and its tests

Two files: the contract, and the experiments that check it

discounts.py
def discounted(price, fraction):
    """Discounted price, 2 dp."""
    return round(price * (1 - fraction), 2)

def average(values):
    """Mean; empty list -> 0.0 (our convention)."""
    if not values:      # the empty edge case
        return 0.0
    return round(sum(values) / len(values), 2)
test_discounts.py
from discounts import average, discounted

def test_discounted_ten_percent():
    assert discounted(100.0, 0.10) == 90.0

def test_average_of_a_few_numbers():
    assert average([10, 20, 30]) == 20.0

def test_average_of_one_element():   # one element
    assert average([42]) == 42.0

def test_average_of_empty_is_zero(): # empty input
    assert average([]) == 0.0
Note what the empty-list test really checks: a documented convention. The docstring promised 0.0; the test holds the docstring to it.

Read it bottom-up

A failing test is not a disaster. It is the tool doing its job — pointing at the exact line to fix.
5 · When red appears

Two suspects, one procedure

Suspect 1: the function

The code under test computes the wrong value — the usual case, and the one tests exist to catch.

Suspect 2: the test

Your expectation was wrong. Re-derive the expected answer by hand before touching the code.

Never "fix" a red test by editing the assertion until it passes. First decide which suspect is guilty — with arithmetic, not hope.
A test you adjusted to match the code's output tests nothing. The expected value must come from outside the code.
