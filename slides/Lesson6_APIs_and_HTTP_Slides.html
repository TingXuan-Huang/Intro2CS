0 · Two definitions

An API is a doorway; HTTP is the language spoken through it

API — Application Programming Interface

A doorway a server opens so that other programs — not humans clicking a website — can ask it for data in a predictable shape.

A website is for people. An API is for code.

HTTP — HyperText Transfer Protocol

The language spoken through that doorway: a fixed grammar for what a request looks like, what a response looks like, and what the status numbers mean.

A protocol is a contract two machines agree on in advance.

your code (client) ──GET series DGS10──▶ FRED's server your code (client) ◀──200 + JSON body─── FRED's server
A question and an answer, both written as text, in a shape both sides agreed on before either spoke.

1 · Client, server, request, response

Two programs, two roles

The client

Starts the conversation by sending a request. Your browser is a client — and so is requests.get.

The client always speaks first.

The server

Listens, does the work, and sends back a response: a status code and, usually, a body.

The server never calls you; it only answers.

Nothing more mystical than a question and an answer — both written as text, in a shape both sides agreed on in advance. That agreed shape is HTTP.
HTTP is Lesson 1's idea at the level of two machines: the contract is a set of properties both sides promise to honour, so neither has to guess.
1 · Anatomy of a URL

An address where every piece answers one question

http://127.0.0.1:8000 /fred/series/observations ? series_id=DGS10 & file_type=json
└─scheme─┘└───host────┘└────────path───────────┘  └──────────query string────────┘
 protocol   which machine   which resource on it     parameters that refine the ask

Scheme

http vs https — the protocol, and whether it's encrypted.

Host

Which machine to ask — here, your own laptop.

Path

Which resource on that machine: observations.

Query

key=value pairs joined by & that narrow the ask.

The path names a resource; the query string after ? carries the parameters that refine which slice of it you want.

A URL is structured data, not a magic string — and Python has a parser for it, like every other format.
1 · GET vs POST

Two verbs cover almost everything you'll do

VerbMeansParameters go inThink
GETread — fetch data, change nothingthe URL's query string"show me DGS10"
POSTsend — submit data to be stored or processeda request body"here is a new record"

Fetching FRED is always GET

You are reading published data. Every call in this lesson is a GET; its parameters ride visibly in the URL.

Why the distinction matters

A GET can be repeated, cached, and bookmarked safely — it changes nothing. A POST may have consequences, so it is treated with more care.

GET reads and puts its parameters in the URL; POST sends and puts them in a body. That is 90% of what you need.
1 · Headers

Metadata rides alongside the message

Content-Type

What format the body is — e.g. application/json. Tells the client how to read the bytes.

User-Agent

Who is asking — a browser, a script, your client. Some servers use it to decide how to answer.

Retry-After

How long to wait after a rate limit. The server telling you, politely, to slow down (Unit 3).

Headers describe the message; the body is the message. Same distinction as a letter's envelope versus its contents.
Metadata about the message, not the data itself — the envelope, not the letter.
1 · Status codes

Four numbers you will actually meet

CodeNameMeaningYour move

404Not Foundthat resource does not existfix the URL or the id
429Too Many Requestsyou hit a rate limitwait, then retry (Unit 3)
500Server Errorthe server broke, not youretry later, then report

2xx · worked

The request succeeded.

4xx · you asked wrong

Bad id, bad URL, too many calls.

5xx · the server failed

Not your fault; retry later.

The families generalise. A 404 is not a crash — it is a well-formed answer that says no.

2 · API keys

The key lives in the environment, never in the code

Once, in your shell — outside the code

Read it at run time
API_KEY = os.environ.get("FRED_API_KEY")

Why the environment

An API key is a per-user password so the server can enforce rate limits. It is a secret — and Lesson 3 said where secrets go.

  • Kept out of the notebook
  • Kept out of git
  • Never committed or shared by accident
Typing a key into a cell puts it in your git history forever. An environment variable is read at run time and leaves no trace in the code.

2 · Fail loudly

raise_for_status() is Lesson 5's "validate at the door"

Without the check

A 404's body might be {"error_message": ...}. Call .json()["observations"] on it and you get a confusing KeyError far downstream — or worse, silently wrong data.

With the check

raise_for_status() turns any 4xx/5xx into a loud, immediate exception — named at the door, before a single bad value enters your pipeline.

Same move as Lesson 5's load_validated: refuse to continue the instant the contract is broken, rather than limping on with bad input.
A response object is guilty until proven 200. Check first; parse second.
2 · JSON carries no types

What is wrong with the first observation?

The first observation arrives as {'date': '2021-01-04', 'value': '0.93'}. Two things in it are not what a DataFrame needs. What are they?
The value is a string, not a number — '0.93', with quotes. And missing days arrive as the string ".", not as a blank or a NaN. JSON, exactly like a CSV, carries no types: everything is text until you say otherwise. This is Lesson 2's situation, arriving over a wire instead of from a file.
The wire drops the same property a file drops: type. Loading a response is Lesson 2's job, again.

2 · Checkpoint

JSON from a wire, CSV from a file — same problem?

You already knew how to load a messy CSV in Lesson 2. What, if anything, is genuinely new about loading a JSON API response — and what is exactly the same?

3 · The rough edges

The happy path is the exception on real networks

Rate limits

The server caps how often you may call. Cross it → 429. Wait and retry.

Pagination

A huge result arrives one page at a time. Loop until you have them all.

Timeouts

A hung server can freeze you. Bound every call with timeout=.

Caching

Fetch once, save the response, read the copy next time. Offline forever.

One request, a 200, clean JSON — that was Unit 2's demo. Four rough edges separate a toy script from a client you can rely on.
None of these is exotic. Each is a small, standard habit that turns a fragile fetch into a durable one.
3 · Rate limits

429 means slow down — don't give up, don't hammer

Why servers rate-limit

A server protects itself by capping how often any one caller may hit it — FRED allows roughly 120 requests per minute. Cross the line and it answers 429 Too Many Requests.

The polite response

Not giving up, and not retrying instantly in a tight loop (which makes it worse). Instead: wait, then retry — backing off a little longer each time.

Our /flaky route returns 429 twice, then 200, so we can watch it.

A 429 is not a failure to swallow — it is the server asking you, in the status vocabulary, to come back in a moment.
3 · Exponential backoff

Wait 0.1s, then 0.2s, then 0.4s — each retry a little longer

def get_with_retries(url, max_attempts=5, base_delay=0.1):
    for attempt in range(max_attempts):
        reply = requests.get(url, timeout=10)
        if reply.status_code != 429:
            return reply             # success (or real error) → stop
        wait = base_delay * (2 ** attempt)  # 0.1, 0.2, 0.4 …
        time.sleep(wait)              # exponential backoff
    return reply
attempt 1: got 429, backing off 0.10s
attempt 2: got 429, backing off 0.20s
final status: 200 →
  {'ok': True, 'served_on_attempt': 3}

2 ** attempt doubles the wait each round. Only a 429 retries; a 200 or a real error (say 404) returns immediately.

Backoff spreads your retries out so a busy server can recover — instead of a thundering herd of instant re-requests.

3 · Pagination

Ask for pages, not floods

Why pages exist

A server will not hand you a million rows in one response — it would be slow and fragile. It returns a page at a time.

The two knobs

offset — where to start. limit — how many rows. Ask for a window, then advance the offset until you've collected all count rows.

0–499page 1
500–999page 2
1000–1438page 3
It is Lesson 4's linear scan, one page per step — the same shape of work, chunked to fit the wire.
3 · Pagination in code

A loop that advances the offset until count is reached

def fetch_all_pages(series_id, page_size=500):
    collected, offset = [], 0
    while True:
        body = requests.get(url, params={
            "series_id": series_id,
            "offset": offset, "limit": page_size},
            timeout=10).json()
        collected.extend(body["observations"])
        offset += page_size
        if offset >= body["count"]: break
    return collected
page at offset    0: 500 rows (total 500)
page at offset  500: 500 rows (total 1000)
page at offset 1000: 439 rows (total 1439)
reassembled 1439 rows from pages

The last page returns fewer than page_size rows — that, plus offset >= count, is how the loop knows it's done.

The stopping condition is the whole game: advance until the offset passes the total the server reported.

3 · Timeouts

Never wait forever

try:
    requests.get(
        f"{LOCAL_FRED}/slow",   # the route sleeps 0.5s
        timeout=0.2,
    )
except requests.exceptions.Timeout:
    print("caught — pipeline stays responsive")

Why every call needs a bound

A request with no time limit can freeze your whole program on a hung server — waiting forever, doing nothing.

timeout= turns "hang forever" into a Timeout exception you can catch and handle — a decision, not a lockup.

Always pass timeout=. A raise is something you can recover from; a hang is not.
3 · Caching to disk

Fetch once; depend on the copy

The move every later lesson reuses

Fetch once, save the response, and next time read the saved copy instead of hitting the network. It is what keeps this whole course runnable offline.

Which format? Lesson 2 answered

Store the raw JSON the server sent. It preserves the response's exact nested shape and re-parses to an identical dict — a cached hit is indistinguishable from a live fetch.

Parquet wins for a big tidied table; a raw API response is a nested document, and JSON is what it already is.

The cache stores the response, not your interpretation of it — so tomorrow's re-read is faithful to what the server actually said.

3 · Checkpoint

Four habits, one reliable client

You call an API 10,000 times in a loop and it starts returning 429s halfway through. Which of the four rough-edge habits saves the run — and which one keeps the next run fast?
Retry with backoff saves this run: on each 429, wait (a little longer each time) and try again instead of crashing or hammering. Caching keeps the next run fast: every response already fetched is on disk, so a re-run skips the network — and the parts you never reached are the only ones that hit the API. (Pagination bounds how much each call returns; timeouts stop any single call from hanging the whole loop.)
Retry, paginate, time out, cache — together they turn "it worked on my laptop once" into "it runs anywhere, repeatedly."

4 · Tidy to monthly

Restore types, then collapse to one row per month

def tidy_monthly(payload):
    frame = pd.DataFrame(payload["observations"])[["date", "value"]]
    frame["date"]  = pd.to_datetime(frame["date"])
    frame["value"] = pd.to_numeric(frame["value"], errors="coerce")
    frame = frame.dropna(subset=["value"])
    monthly = (frame.set_index("date")["value"]
                    .resample("MS").mean()   # one row per month
                    .dropna().reset_index())
    monthly.columns = ["month", "value"]
    return monthly

resample("MS") groups by month-start; .mean() is Lesson 4's reduce. The functional-pipeline shape of Lesson 5, on fetched data.

4 · SDK vs raw HTTP

One honest paragraph about the convenience layer

What an SDK gives you

fred.get_series("UNRATE")

One line, no URL to build. For FRED there's fredapi and pandas-datareader. Genuinely lovely when it fits.

What it costs

  • It can lag the API's newest features.
  • It hides which parameters exist.
  • It's a dependency you must trust and update.
Same Lesson 4 lesson as max() over the hand-written loop: the convenience runs the procedure for you; it never removes it. An SDK is a wrapper over the exact GET you just wrote.
Learn the request first. Reach for the SDK once you know what it's doing on your behalf — then you can debug it when it lags.

The lesson, one sentence

An API is a doorway and HTTP is the contract: send the GET, check the status, restore the types — and because the network is unreliable, retry, paginate, time out, and cache.

The data you fetched once is yours to keep. The same move as every earlier lesson: make the properties explicit.
