1 · What an LLM is

One repeated move

given the text so far → predict the next token → append it → predict again

That is the whole engine — Lesson 4 would call it an algorithm. Answering questions, writing code, classifying headlines: all of it is that one move, run over and over.

Tokens

the subword chunks the model actually reads — and the unit you pay in

Next-token prediction

the single move, repeated until a stop

Context window

the bounded amount of text it sees at once

Temperature

how adventurously the next token is sampled

Four words make the rest of the lesson make sense. Unit 1 pins each one down with running code.
1 · Tokens

The model doesn't see characters or words

def estimate_tokens(text):
    """Rough English rule of thumb: ~4 characters per token.
    The AUTHORITATIVE count comes from the model's tokenizer,
    or the API's usage field — this is only for budgeting."""
    return max(1, round(len(text) / 4))

sample = "Unemployment falls to 3.9% as hiring accelerates"
print("characters:", len(sample),
      "| estimated tokens:", estimate_tokens(sample))
characters: 48 | estimated tokens: 12

It sees tokens: subword chunks like  inflation, ation,  3, .9. The exact split comes from the model's tokenizer, not from us — we can't run it offline, so we estimate and stay honest that it's an estimate.

~4 characters per English token is a budgeting rule of thumb — good enough to forecast a cost, never good enough to bill by.
1 · Checkpoint

Estimate vs authority

Our function said the headline is ~12 tokens. Where do the authoritative counts come from — and why does it matter that we can name the difference?
The authoritative counts come from the model's tokenizer, and — for any call you actually make — from the response's usage block, which reports the exact input and output tokens you were billed on. The estimate is for budgeting before you spend; the usage field is the invoice. Confusing the two is how a forecast quietly becomes a claim — the same honesty rule as Lesson 2's "a snapshot is dated data, not live data."
Estimate to plan; read usage to know. Keep the two labelled.
1 · A toy model you can read

A bigram model: count which word follows which

import random
from collections import defaultdict, Counter

corpus = """
inflation cooled again as consumer prices rose modestly
consumer prices rose again as inflation stayed high
the central bank held interest rates steady
the central bank raised interest rates again
markets rose as the central bank held rates steady
""".lower().split()

# Count which word tends to follow which (a "bigram" model).
following = defaultdict(Counter)
for first, second in zip(corpus, corpus[1:]):
    following[first][second] += 1
following["the"]   → {'central': 3}
following["bank"]  → {'held': 2,
                      'raised': 1}
following["rates"] → {'steady': 2,
                      'again': 1}

A real model weighs thousands of tokens of context and was trained on trillions — but the move is the same, and this toy fits on one slide.

"What tends to follow what," learned by counting. That table is the toy's entire knowledge of the world.
1 · Generation and temperature

Pick the next word — greedily, or by sampling

def next_word(word, temperature, rng):
    choices = following[word]
    if not choices:
        return None
    words = list(choices)
    counts = [choices[w] for w in words]
    if temperature <= 0:
        return words[counts.index(max(counts))]      # greedy
    weights = [c ** (1.0 / temperature) for c in counts]  # flatter odds
    return rng.choices(words, weights=weights, k=1)[0]

def generate(start, temperature, seed, length=8):
    rng = random.Random(seed)
    out = [start]
    for _ in range(length):
        nxt = next_word(out[-1], temperature, rng)
        if nxt is None: break
        out.append(nxt)
    return " ".join(out)

The two regimes

Temperature 0 — greedy. Always take the single most likely next word: deterministic, the same every run.

Temperature high — sample. Less likely continuations get a real chance: more variety, and more risk.

c ** (1.0 / temperature) flattens the odds as temperature rises — the counts matter less and less.

Generation is a loop around one function. Everything the model "says" is drawn, word by word, from what tended to follow what.

1 · Hallucination

Not a bug to be patched — the table's bottom row, stated plainly

When plausible and true coincide

The model emits the most plausible next tokens, and those happen to be correct. It looks like knowledge — and for well-trodden facts, it usually is.

When they diverge

You get fluent, confident, wrong text — with the same tone, the same grammar, the same self-assurance as the correct answers around it.

"Plausible next token" is simply not the same property as "true." The toy on the last slide never checked one word against reality — a real model doesn't either.
Everything in Unit 4 is about living with that. Unit 1's takeaway: useful output, shipping with no guaranteed properties — truth least of all.
1 · Checkpoint

Name the two properties

A hallucination is the direct consequence of two properties not being the same. Which two — and why does calling it a "rare glitch" get the mental model wrong?
"Plausible next token" and "true." The model optimizes the first; nothing in its one repeated move ever checks the second. So hallucination isn't a malfunction that occasionally interrupts correct behavior — it is the normal mode, observed in the cases where plausible and true happen to diverge. A glitch you patch; a property gap you engineer around — which is exactly what Units 3 and 4 do.
Unit 1 gave you the vocabulary. Unit 2 makes the call — the same REST request you built in Lesson 6.

2 · System vs user

The rules of the game vs the move

System — the rules

Frames everything that comes after it: the persona, the output format, the constraints. Written once, reused for every call in a job.

User — the move

The specific item to handle now: this headline, this sentence, this question. Changes on every call; the rules don't.

The same words behave differently in each slot. "Reply with JSON only" as a system rule governs the whole run; buried in a user message it competes with everything else in that one turn.
Get the framing right once, reuse it for every call — that split is what makes a 20-headline classification job one system prompt plus 20 tiny user messages.
2 · Via requests

Lesson 6's pattern, pointed at a new endpoint

import requests

ANTHROPIC_URL = "https://api.anthropic.com/v1/messages"
MODEL = "claude-haiku-4-5-20251001"   # small, cheap — a fine default for classification

def call_via_requests(messages, system=None, max_tokens=200):
    headers = {
        "x-api-key": os.environ.get("ANTHROPIC_API_KEY", ""),
        "anthropic-version": "2023-06-01",
        "content-type": "application/json",
    }
    body = {"model": MODEL, "max_tokens": max_tokens, "messages": messages}
    if system is not None:
        body["system"] = system
    response = requests.post(ANTHROPIC_URL, headers=headers, json=body, timeout=30)
    response.raise_for_status()
    return response.json()

Count the new pieces

  • the endpoint URL
  • two headers: x-api-key, anthropic-version
  • the body shape: model, max_tokens, messages

Everything else — timeout=, raise_for_status(), .json() — is Lesson 6, verbatim.

Defined for reference; we don't call it live in class. The point is that you could — you already know every line.
2 · Key hygiene

The key lives in the environment — same rule, higher stakes

Never

  • pasted into a notebook cell — it lands in git history forever
  • pasted into a prompt — it leaves your machine as data
  • printed in output — screenshots and logs travel

Always

os.environ.get("ANTHROPIC_API_KEY", "")

Read at run time, absent from the code, invisible to git. Lesson 6's FRED_API_KEY rule, unchanged.

Higher stakes than FRED: this key is attached to a bill. A leaked FRED key costs you rate limits; a leaked Anthropic key spends someone's money.
Secrets live outside the code. The environment variable is the only door the key walks through.
2 · The SDK equivalent

Two doors, one room

def call_via_sdk(messages, system=None, max_tokens=200):
    """The same call through the anthropic SDK. Under the
    hood it POSTs to the same URL with the same headers."""
    client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY from the environment
    kwargs = {"model": MODEL, "max_tokens": max_tokens,
              "messages": messages}
    if system is not None:
        kwargs["system"] = system
    return client.messages.create(**kwargs)

Convenience over the raw POST

No hand-built headers, no URL string. Like pd.read_csv over parsing a file by hand in Lesson 2 — and like Lesson 6's SDK verdict: the wrapper runs the request for you; it never removes it.

Both doors send the identical request. Pick whichever reads better to you.

Learn the raw request once, then let the SDK type the boilerplate — you can now debug what it does on your behalf.
2 · Running offline

Record once, replay forever

The mechanism

A recorded response is just the API's JSON, saved to disk in its exact wire shape. Replaying it is json.loads on a file — no key, no network, no bill, and the same result every run.

The honesty label

Our fixtures are authored teaching examples written by hand in the real wire format — not captures of live calls. Token counts are plausible estimates. The label matters: provenance is a property, and we track it like any other.

Where have you seen this move before? Lesson 2's pinned data files, Lesson 6's offline FRED stand-in — pin the input, and code becomes testable and class becomes portable.
One entry point, call_llm(), decides: key set → live SDK call; no key → replay the named fixture and say so.

def call_llm(messages, system=None, fixture=None, max_tokens=200):
    if HAVE_KEY:
        message = call_via_sdk(messages, system=system, max_tokens=max_tokens)
        return message.model_dump() if hasattr(message, "model_dump") else message
    if fixture is None:
        raise RuntimeError("no API key set and no fixture named")

    return json.loads((FIXTURE_DIR / f"{fixture}.json").read_text())

3 · Structured output

"Usually JSON" is not a property you can build on

Why JSON at all

For a program to use a model's answer, the answer needs a shape a program can parse. So we ask for JSON — a category and a confidence, nothing else.

The catch

Asking for JSON gets you JSON usually. Real models mostly comply — and sometimes don't. "Usually" is exactly the kind of promise this course has taught you to refuse.

The whole unit is one habit: validate every reply, and retry the ones that fail. Lesson 5's validate-early, aimed at a source that can hand you anything.
You cannot make the model reliable. You can make your pipeline reliable around it — check, then retry.

3 · Two gates

Gate 1: does it parse? Gate 2: is it the right JSON?

def parse_json(text):
    try:                                   # Gate 1 — is it JSON at all?
        return json.loads(text)
    except json.JSONDecodeError:
        return None

VALID_CATEGORIES = {"monetary_policy", "inflation", "employment",
                    "markets", "trade", "housing"}

def is_valid(payload):                     # Gate 2 — the right JSON?
    if not isinstance(payload, dict):
        return False
    if set(payload) != {"category", "confidence"}:
        return False
    if payload["category"] not in VALID_CATEGORIES:
        return False
    confidence = payload["confidence"]
    if isinstance(confidence, bool) or not isinstance(confidence, (int, float)):
        return False
    return 0.0 <= confidence <= 1.0

Lesson 1's checklist, as code

  • exactly the keys {"category", "confidence"}
  • category drawn from the legal set — set membership
  • confidence a real number in 0..1

Gate 1 returns None instead of raising; Gate 2 returns False instead of crashing. A validator that crashes is no safer than no validator.

Two small, testable functions — Lesson 5's shape. Every reply passes through both gates or it does not count.
3 · The gates, run

Four recorded replies, two verdicts each

Read the pattern

The fenced and prose replies fail Gate 1 — the ``` and the "Sure!" break json.loads. The missing-key reply passes Gate 1 and fails Gate 2.

Both gates are necessary; neither alone is enough.

An LLM reply has no property until you check it — not even "is JSON." Same discipline as Lesson 5's validate-at-the-boundary; the boundary just leaks in more ways now.
One clean reply out of four. That ratio is authored for teaching — but every failure kind in it happens in production.
3 · Checkpoint

Why does is_valid reject bool by name?

The confidence check reads if isinstance(confidence, bool) or not isinstance(confidence, (int, float)). Why single out bool — wouldn't the (int, float) check already reject it?
No — in Python, bool is a subclass of int: isinstance(True, int) is True, and 0.0 <= True <= 1.0 is also True. Without the explicit exclusion, {"category": "markets", "confidence": true} would sail through both gates and put a boolean where your pipeline expects a probability. Property checks have to be written against the sneaky cases, not the obvious ones — Lesson 1's types, with their real edge behavior.
A schema check is only as strong as its weirdest counterexample. The exercise's checker probes exactly this.

3 · Retry on invalid

The model is stochastic — so ask again, a bounded number of times

RETRY_SEQUENCE = ["classify_fenced", "classify_clean"]  # attempt 0 bad, attempt 1 good
MAX_ATTEMPTS = 3

def classify_with_retry(headline, sequence):
    for attempt in range(MAX_ATTEMPTS):
        fixture = sequence[min(attempt, len(sequence) - 1)]
        text = text_of(call_llm([{"role": "user", "content": headline}],
                                system=CLASSIFY_SYSTEM, fixture=fixture))
        payload = parse_json(text)
        if payload is not None and is_valid(payload):
            return {"payload": payload, "retries": attempt}
        print(f"  attempt {attempt}: invalid ({fixture}) -> retrying")
    return {"payload": None, "retries": MAX_ATTEMPTS - 1}

   'confidence': 0.95},

Temperature made the model a random draw — so the next draw is often clean. Bounded means it tries a few times, then gives up rather than looping forever.

Lesson 6 retried a flaky network (429 → back off). Here you retry a flaky answer — same shape, different failure.

3 · The retry count is data

Keep the number — it's telling you something

Low retry rate

Your prompt and schema are working; the model usually complies on the first draw.

High retry rate

A signal your prompt needs work — or your model is too small for the task. Fix the cause, don't just pay for more retries.

Failures after retries

Some items never validate within the budget. Report them as failed — never smuggle them through.

The loop turned an unreliable source into a reliable one — not by trusting harder, but by checking and retrying. Measuring exactly that is the heart of this lesson's exercise.
A retry loop that doesn't count its retries throws away the best diagnostic it produces.
3 · The two workhorses

Classification and extraction — same ending, different property

EXTRACT_SYSTEM = (
    "Extract the fields from the sentence. Reply with ONLY JSON: "
    '{"company": <string>, "metric": <string>, "value_pct": <number>}.'
)
sentence = "Acme Corp said quarterly revenue rose 12.5% from a year earlier."
payload = parse_json(text_of(call_llm(
    [{"role": "user", "content": sentence}],
    system=EXTRACT_SYSTEM, fixture="extract_acme")))

def extraction_is_valid(p):
    return (isinstance(p, dict)
            and set(p) == {"company", "metric", "value_pct"}
            and isinstance(p["company"], str)
            and isinstance(p["metric"], str)
            and isinstance(p["value_pct"], (int, float))
            and not isinstance(p["value_pct"], bool))
extracted: {'company': 'Acme Corp',
  'metric': 'revenue',
  'value_pct': 12.5}
valid extraction: True

Classification → fixed labels; enforce set membership. Extraction → structured fields from messy text; enforce schema shape and field types.

Almost every practical LLM data task is one of these two shapes — and both end the same way: a JSON object you validate before you trust.
3 · Checkpoint

Passing Gate 1 is not passing

A reply passes json.loads cleanly. Name three distinct ways it can still be unusable — and the habit from earlier lessons that catches all three.
It can have the wrong keys (missing confidence, or an extra field you never asked for); a category outside the legal set ("unemployment" instead of "employment" — plausible, and wrong); or an out-of-range or mis-typed value (confidence 1.4, or the boolean true). The habit is Lesson 5's validate at the boundary: a second gate that checks the parsed object against the property checklist — exactly what is_valid does, and exactly what you implement in the exercise.
Unit 3 gave the reply a shape. Unit 4 decides what to send, believe, and spend.
4 · Trust boundaries

The model works. Three boundaries keep it from hurting you

What you send

Privacy. Your prompt leaves your machine and is processed by a third party. Send the smallest slice of data that lets the model do the job.

What you believe

Verification. A model predicts plausible tokens. Any number you can compute exactly in Python, compute in Python.

What you spend

Cost. Tokens are money. Pick the smallest capable model, send fewer tokens, cap the reply, forecast before you run.

Unit 4 is the discipline layer. None of it is exotic — each boundary is one small habit applied every time.
4 · What you send

Minimize: the task needs the note, not the customer

raw = {"customer_id": "00101",
       "email": "alice@example.com",
       "note": "asking about a refund on order T007"}

# The task only needs the free-text note — so send only that.
to_send = {"note": raw["note"]}
print("on your machine :", raw)
print("sent to the API :", to_send)
on your machine : {'customer_id':
  '00101', 'email':
  'alice@example.com', 'note':
  'asking about a refund on order T007'}
sent to the API : {'note': 'asking
  about a refund on order T007'}
Treat a prompt like an email to a vendor: no secrets, no credentials, no personal data you aren't cleared to share.
The model can't leak what it never received — the privacy version of Lesson 2's "load only the columns the question needs."
4 · What you believe

Never let the model do arithmetic you can do exactly

numbers = [1200, 950, 1400, 1800, 1250]
question = f"What is the sum of {numbers}? Reply with just the number."

model_answer = text_of(call_llm(
    [{"role": "user", "content": question}], fixture="bad_sum"))
python_answer = sum(numbers)

print("model says :", model_answer)
print("python says:", python_answer)
print("agree?     :", str(python_answer) in model_answer)
model says : 6400
python says: 6600
agree?     : False

Fluent, confident, and off by 200. "The sum of these numbers" is a plausible-looking number — that's all the model ever promised.

If a number matters, compute it. Sums, counts, sorts, lookups — Python does them exactly, for free.
4 · The division of labor

Python computes; the model phrases

Python's side

The exact algorithms: sums, sorts, matches, joins, aggregations. Verified, deterministic, auditable — everything Lessons 2–9 built.

The model's side

Language. Turning your verified 6600 into "Sales totaled $6,600 across the five months." Classifying, extracting, summarizing — text work.

Lesson 4's closing rule, verbatim: let Python run the exact algorithms and let the model interpret the verified results — never the reverse.
The model is a language layer over your computation, not a replacement for it.
4 · What you spend

Three habits, one sane bill

Smallest capable model

Unit 2's table: a ~5× swing on the same job. Start with Haiku; escalate only when the task proves it needs more.

Fewer tokens

Short prompts, minimal data — the privacy habit and the cost habit are the same habit.

Cap max_tokens

A runaway reply can't run up the cost if the cap cuts it off. You saw the knob in every call signature.

cheapest model, per scale (Haiku 4.5, dated snapshot):
          60 lines  ~  $      0.01
      60,000 lines  ~  $      7.30
   6,000,000 lines  ~  $    730.40

Estimate on a sample, then multiply — a forecast you can make before spending a cent, because cost is linear in tokens.

Same "measure on small, reason about big" move as Lesson 4 — except here it maps straight to dollars.
4 · Why the whole lesson ran offline

Pinned responses are how you test LLM code

The problem with testing against a live model

Nondeterministic output, a network dependency, and a bill on every run. A test that sometimes fails and always costs money is a test nobody runs.

The fixture answer

Replay pinned responses: deterministic inputs, so you can check your validation and retry logic — the code that is actually yours — without paying, waiting, or depending on the network.

Notice what's being tested: not the model, your handling of it. The malformed fixtures exist precisely so your gates get exercised.
The exercise is built entirely this way — Lesson 5's checker discipline, pointed at LLM plumbing.
4 · Checkpoint

One prompt, three boundary crossings

A teammate pastes an entire customer table into a prompt and asks the model for the average order value. Name every trust boundary that crosses — and the fix.
Send: the whole table leaves the machine — emails, IDs, everything the question didn't need. Believe: "the average" will come back as a plausible number, unverified — arithmetic belongs in Python. Spend: a full table is thousands of input tokens for a job that needed zero. The fix: compute df["order_value"].mean() in pandas, send nothing — or, if a written summary is wanted, send only the verified number and let the model phrase it.
All three boundaries, one everyday mistake. The discipline is knowing which tool owns which step.
