1 · The pain of files

Bytes cannot make a promise

A file's strength

A file is bytes plus an agreed encoding — universal, portable, readable by anything. That is why CSV conquered the world.

And its whole weakness

Bytes cannot insist. A CSV cannot demand that a column stays text, that an id never repeats, that a country is never blank, or that two files agree with each other.

Every failure in this unit is a property from Lesson 1 that the file failed to keep. We line three of them up on purpose — because Unit 2 is where we finally get to enforce each one.
Lesson 2 met these failures one at a time and repaired them. Today we ask why the repair was ever necessary.
1.1 · The leading-zero trap

Write 00123, read back 123

people = pd.DataFrame({"customer_id": ["00123", "00204", "00987"],
                       "spend": [40, 55, 12]})

trap_path = Path("../data/lesson7_leading_zero.csv")
people.to_csv(trap_path, index=False)

print(back)
print("dtype:", back["customer_id"].dtype)
   customer_id  spend
0          123     40
1          204     55
2          987     12
dtype: int64

The file on disk really says 00123 — you can open it and see the zeros. It is the read that destroyed them: all digits, so pandas guessed int64.

Persistence without a promise: the value survived, the property — "this is a label, not a quantity" — did not.

1.1 · Checkpoint

Did the file lie to us?

We wrote 00123; we got back the number 123. Was the file at fault — and what was Lesson 2's fix, exactly?
The file did not lie — it faithfully stored the characters 00123. It simply has no place to record that customer_id is a label, not a quantity, so the reader had to guess. Lesson 2's fix was dtype={"customer_id": "string"} on the way in — a repair, applied by whoever remembered to apply it. Nothing on disk forces the right reading, on any read, by anyone.
The bug is not in pandas. It is in a format that cannot carry the property — and can therefore never enforce it.

1.2 · The diagnosis

A file is an island

What just happened

Neither file can say "there is exactly one true country per customer, and it lives here." A file knows nothing about any other file — it cannot even see the contradiction, let alone refuse it.

What we want instead

A single place that owns the customer record and refuses to hold two. One customer_id → one row, and any second row with the same id is rejected at the door.

That place is a table with a primary key — the first thing we build in Unit 2.

The cure for "two truths" is not a cleverer merge. It is one table that is the truth.

1 · What persistence really means

"I saved a file" is one third of the job

Durability

The data outlives the program that made it. A file gives you this — necessary, and nowhere near enough.

Integrity

A referee that checks every write against the rules and rejects violations, so the bad row never lands. Lesson 2 cleaned data; this keeps it from getting dirty.

Coordination

Many readers and writers at once, without two of them clobbering each other's edits.

Persistence plus integrity plus coordination is what the word "database" means.
The rest of this lesson is those three words made concrete — starting with the referee.
1 · Checkpoint

Name what the file could not do

Three demos, three failures. For each one — the vanished zeros, the two countries, the blank country — name the Lesson 1 property that was lost, and say what the file offered instead of enforcement.
Type: 00123 became 123 — the file cannot record "label, not quantity"; it offered a re-declared dtype= repair on every read. Uniqueness: two files held two countries for one id — no single owner of the record; it offered a merge and an argument. Required-ness: a blank country landed on disk — no syntax for "required"; it offered isna() archaeology afterwards. In every case: durability without integrity.
Unit 2 writes the refusal down in an executable language: CREATE TABLE.
2 · Tables, schemas, keys

The property checklist, in a language software enforces

The vocabulary

  • Table — rows that all share one declared shape.
  • Schema — that declared shape: the columns and the rules.
  • Key — the column whose value identifies a record.

The one idea of this unit

When you CREATE TABLE, you are not only naming columns — you are declaring the rules and handing the database the authority to reject anything that breaks them.

A schema is Lesson 1's checklist with a referee attached. Everything a file could not refuse in Unit 1, the schema refuses in this one.
You stop hoping the rules hold. You declare them, and the engine says no on your behalf.
2 · Opening a database

Connect — and flip one crucial switch

DB = Path("../data/lesson7_demo.sqlite")
if DB.exists():
    DB.unlink()            # start clean so re-running is repeatable

conn = sqlite3.connect(DB) # opens (or creates) the database file

# SURPRISE: SQLite ships with foreign-key checking turned OFF,
# per connection. Switch it on every time, or the relationship
# rules in 2.3 are silently ignored.
conn.execute("PRAGMA foreign_keys = ON")
database file : ../data/lesson7_demo.sqlite
foreign keys  : 1 (1 = enforced)

Remember this PRAGMA. It is per connection — a habit, not a one-time setting. We return to it when a rule mysteriously fails to fire.

connect gives you a live database in one line. The file did not even have to exist yet.
2.1 · CREATE TABLE

Read this statement as a list of Lesson 1 properties

conn.execute("""
    CREATE TABLE customers (
        customer_id TEXT PRIMARY KEY,   -- unique label, never a number
        country     TEXT NOT NULL,      -- required: no blank countries
        signup_date TEXT                -- optional
    )
""")

conn.execute(
    "INSERT INTO customers (customer_id, country, signup_date) VALUES (?, ?, ?)",
    ("00123", "United Kingdom", "2010-12-01"),
)
conn.commit()

Line by line

  • TEXT PRIMARY KEY — a label (so text; leading zeros are safe) that must be unique.
  • TEXT NOT NULL — required; the blank from 1.3 is now illegal.
  • TEXT alone — optional; no promise made, so none enforced.
That CREATE TABLE is the checklist. From now on the database — not your memory, not a comment, not a downstream assert — guarantees those rules.
2 · Three promises

Three constraints, kept on every write, forever

PRIMARY KEYexactly one row per key value — a second row with the same customer_id is rejected, not merged, not appended.
NOT NULLthe column can never be blank — an insert with a missing country is rejected before it lands.
FOREIGN KEYthe value must already exist in the referenced table — a transaction naming an unknown customer is rejected as an orphan.
Each promise is enforced by refusal at write time — the exact move Unit 1's files could not make.
2.2 · The heart of the lesson

The database rejects a bad row — loudly, specifically, instantly

# (a) A DUPLICATE primary key -- "two countries" from 1.2.
try:
    conn.execute("INSERT INTO customers VALUES (?, ?, ?)",
                 ("00123", "France", "2011-01-01"))
except sqlite3.IntegrityError as err:
    print("duplicate customer rejected ->", err)

# (b) A BLANK required column -- the missing country from 1.3.
try:
    conn.execute("INSERT INTO customers VALUES (?, ?, ?)",
                 ("00204", None, "2011-02-01"))
except sqlite3.IntegrityError as err:
    print("empty country rejected      ->", err)

2.2 · Checkpoint

How is that rejection so cheap?

customers could hold ten million rows. How does the database decide, instantly on every insert, whether 00123 is already taken — and where have you built that structure yourself?
It keeps an index of the keys it has already seen and asks "is 00123 in here?" — Lesson 4's O(1) membership test, now durable. The set-of-seen-keys you built by hand to speed up a join is the very structure a primary key maintains for you permanently — as a rule rather than an optimisation.
A primary key is a dictionary lookup with authority: same speed, plus the right to say no.
2.3 · FOREIGN KEY

A merge is hopeful; a foreign key is a guarantee

Lesson 2's merge

We linked two tables by customer_id and merged. But a merge is hopeful: it happily kept transaction T006, whose customer 00999 did not exist, as a row full of NaN.

The enforced relationship

A FOREIGN KEY declares: a transaction may only name a customer that actually exists. The orphan is not analysed afterwards — it is refused at the door.

The difference in one sentence: analysing a broken relationship versus forbidding it.
The merge key you trusted in Lesson 2 becomes a rule the engine enforces on every insert.
2.3 · FOREIGN KEY in code

One accepted transaction, one rejected orphan

conn.execute("""
    CREATE TABLE transactions (
        transaction_id TEXT PRIMARY KEY,
        customer_id    TEXT NOT NULL,
        amount         REAL NOT NULL,
        FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
    )
""")

# A transaction for a customer who exists -> accepted.
conn.execute("INSERT INTO transactions VALUES (?, ?, ?)", ("T001", "00123", 9.95))

# A transaction for customer 00999, who is NOT in customers -> rejected.
try:
    conn.execute("INSERT INTO transactions VALUES (?, ?, ?)", ("T002", "00999", 5.00))
except sqlite3.IntegrityError as err:
    print("orphan transaction rejected ->", err)
orphan transaction rejected ->
  FOREIGN KEY constraint failed

transactions that made it in:
  transaction_id customer_id  amount
0           T001       00123    9.95
The exact row a left join would have kept as NaN, the database threw out at the door.

2.3 · The gotcha

Foreign keys are OFF until you switch them on

The silent failure mode

SQLite ships with foreign-key checking disabled, per connection. Skip the switch and the orphan insert succeeds — no error, no warning, a broken relationship stored as if it were fine.

The habit: connect, then switch, every time
conn = sqlite3.connect(DB)
conn.execute("PRAGMA foreign_keys = ON")

The exercise checker verifies this on your returned connection — test_foreign_keys_are_enforced_on_the_connection exists precisely because this bite is so quiet.

A constraint you never switched on is documentation, not enforcement. The rule is only as good as the switch.
2.4 · One honest caveat

Types are affinities — not always walls

conn.execute("CREATE TABLE loose (n INTEGER)")
conn.execute("INSERT INTO loose VALUES ('banana')")   # accepted!
print("plain table stored:", conn.execute("SELECT n FROM loose").fetchone())

conn.execute("CREATE TABLE strict_demo (n INTEGER) STRICT")
try:
    conn.execute("INSERT INTO strict_demo VALUES ('banana')")
except sqlite3.IntegrityError as err:
    print("STRICT table rejected the wrong type ->", err)
plain table stored: ('banana',)
STRICT table rejected the wrong type ->
  cannot store TEXT value in
  INTEGER column strict_demo.n

SQLite is strict about NOT NULL, PRIMARY KEY, FOREIGN KEY — and deliberately lenient about column types, unless you declare the table STRICT.

A schema enforces most of Lesson 1's checklist out of the box — and the rest (exact types) when you ask for STRICT.
2 · Checkpoint

Walls and suggestions

Which of Lesson 1's properties does a default SQLite table enforce as hard walls — and which one is only a suggestion until you say otherwise?
Hard walls: uniqueness (PRIMARY KEY), required-ness (NOT NULL), and relationships (FOREIGN KEY — once PRAGMA foreign_keys = ON is set). Suggestion: the column type — plain SQLite stored 'banana' in an INTEGER column (type affinity). Declaring the table STRICT turns the type into a wall too.

The fix (and the exercise)

Write the CREATE TABLE yourself first, then load into it with to_sql(…, if_exists="append") — rows into your schema, not a guessed one.

The same lesson as CSV amnesia in Lesson 2: a convenient writer drops the properties unless you declare them yourself.

Where have you seen this before?

3.3 · Breaking the rule on purpose

Paste a value into the SQL string and watch it explode

messy_value = "YOU'RE CONFUSING ME METAL SIGN"   # a genuine product name

# WRONG: paste the value straight into the query string.
pasted = f"SELECT * FROM transactions WHERE product = '{messy_value}'"
try:
    retail.execute(pasted).fetchall()
except sqlite3.OperationalError as err:
    print("  -> the database choked:", err)

# RIGHT: pass the value as a parameter; the driver handles the quoting.
safe = retail.execute(
    "SELECT transaction_id FROM transactions WHERE product = ?",
    (messy_value,),
).fetchall()
the string we built:
  SELECT * FROM transactions
  WHERE product = 'YOU'RE CONFUSING
  ME METAL SIGN'
  -> the database choked:
     near "RE": syntax error

parameterized query worked:
  [('536381-02',), ('541509-51',)]
The apostrophe in YOU'RE closed the string early, and the rest became broken SQL. And that is the harmless version.
3.3 · SQL injection

The dangerous version: when the value wants to break the query

A crafted input

A value like '; DROP TABLE customers; -- pasted into a query turns it into two commands — the second of which deletes your table. This is SQL injection, and it has emptied real companies' databases.

The fix — always the same, always easy

Never build queries with + or f-strings. Pass every value through a ? placeholder and let the driver quote it. Data stays data; it can never become code.

The ? is not a style preference. It is the rule — for correctness today (apostrophes) and safety always (injection).
One habit covers both failure modes, and it costs a single character.

3 · Checkpoint

Convince a teammate

A teammate's dashboard builds its SQL with f-strings and "has worked fine for months." What two arguments do you give for switching to ? placeholders — one about correctness, one about safety?
Correctness: any legitimate value containing a quote — a product called YOU'RE …, a name like O'Brien — breaks the pasted string the day it first appears. 4 · The decision guide

When a plain file is genuinely fine — and when it is not

Reach for a CSV / DataFrame when…

  • the data is small and fits in memory, and you are exploring or prototyping;
  • there is one author and no concurrent writers to coordinate;
  • it is a one-off analysis, or a hand-off a human will open in Excel;
  • the properties barely matter because the data is throwaway.

Reach for a database when the data must…

  • outlive the session — a system of record, not a scratchpad;
  • be trusted by many people, reading and writing at once;
  • enforce its own rules — a single bad row is a real problem;
  • grow past memory — query the slice you need, not the whole.
The honest summary: a DataFrame is a fast workbench; a database is a durable system of record.
4 · Checkpoint

Diagnose the shared spreadsheet

A three-person shop keeps its orders ledger in orders.xlsx on a shared drive. Of the three database words — durability, integrity, coordination — which is already failing them, and what breaks next?
Coordination fails first: two people saving at once, and one person's edits silently overwrite the other's. Integrity breaks next: nothing rejects a duplicate order id or an order with no customer, so the errors accumulate quietly. The file only ever gave them durability. A ledger that several people must trust is the textbook case for a real database.
When you hear "we keep it in a shared spreadsheet," you can now name exactly what will go wrong, and in what order.
