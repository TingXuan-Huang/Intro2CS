0 · Algorithm anatomy

A useful algorithm has a contract and a finite procedure

Input

What data is provided, in what representation, under which preconditions?

Procedure

Finite, unambiguous steps that a literal computer can repeat.

Output

What result should be returned, and what must be true about it?

Input: a nonempty list of comparable sales values Procedure: examine the values while carrying the best seen so far Output: one value equal to the maximum input value
Correctness comes first: the output must satisfy the contract for every permitted input.
1 · Maximum

Maximum is a scan carrying the best value seen so far

def find_max(values):
    if not values:
        raise ValueError("empty input")

    best = values[0]
    for value in values[1:]:
        if value > best:
            best = value

    return best
InitializeThe first value is best among the first one.
CompareAsk whether the next value is better.
UpdateReplace best only when the comparison succeeds.
ReturnAfter the scan, best summarizes the whole list.
The carried-state invariant is: best equals the maximum of the values processed so far.

1 · Minimum

Minimum reuses the same pattern with one changed comparison

def find_min(values):
    if not values:
        raise ValueError("empty input")

    smallest = values[0]
    for value in values[1:]:
        if value < smallest:
            smallest = value

    return smallest

The structure is unchanged

Initialize from a valid value, scan the rest, compare, update, return.

Only the update rule changes from > to <.

This is algorithm-pattern reuse, not copied intuition.

Small changes in the contract often become small changes in one decision rule.

1 · Edge cases

Edge cases belong in the algorithm contract

What should maximum, minimum, and average do on an empty list? What about one value or tied values?
Empty: maximum, minimum, and average are undefined unless the contract specifies a default; raising ValueError is explicit. One value: all three summaries equal that value. Ties: the value is unambiguous, but returning an index requires a rule such as “first occurrence.”

Empty input

Do not let values[0] or division by zero decide the behavior accidentally.

One item

The loop may run zero additional comparisons and still return a correct result.

Ties

Define whether the output is a value, any position, the first position, or every position.

1 · One-pass summary

Several summaries can share one scan

def summarize(values):
    if not values:
        raise ValueError("empty input")

    smallest = largest = values[0]
    total = 0

    for value in values:
        total += value
        if value < smallest:
            smallest = value
        if value > largest:
            largest = value

    return {
        "min": smallest,
        "max": largest,
        "average": total / len(values),
    }

One traversal, multiple updates

Every value is read once. The loop body updates three independent summaries.

This may reduce passes over slow streams or large data, though the built-ins are often clearer for ordinary code.

2 · Ties and stability

Stable sorting preserves the prior order of equal keys

orders = [
    {"id": "T1", "region": "West", "amount": 50},

    orders,
    key=lambda row: (-row["amount"], row["region"]),

Two useful rules

A tuple key compares the first field, then uses later fields to break ties.

If the keys are equal, Python's stable sort preserves the records' original relative order.

Stability lets earlier ordering decisions remain meaningful after a later sort.

2 · Insertion sort

Insertion sort grows a sorted prefix by inserting the next value

1sorted
3sorted
5sorted
2insert

Shift 5 and 3 right, then place 2 after 1.

Why it is useful

Small or nearly sorted inputs require little shifting.

Reverse-ordered inputs require many shifts.

Like sorting cards in your hand, it maintains a correct sorted prefix after every insertion.

Different sorting procedures can produce the same order while performing very different amounts of work on the same input.

During a merge

Why this changes the work

Every level touches all values, but splitting creates only a small number of levels.

3 · Linear search

Check candidates in sequence and stop on a match

def linear_search(values, target):
    for index, value in enumerate(values):
        if value == target:
            return index
    return None

Contract

Input: values and a target

Output: first matching index, otherwise None

Precondition: none about ordering

An early return improves inputs with an early match, but it cannot help an absent target.

5 · Define the measurement

First define n and the basic operation being counted

Input size, n

Number of sales, rows, nodes, tokens, or another meaningful dimension.

Basic operation

A comparison, lookup, update, or other repeated unit of work.

Cost function, T(n)

How many basic operations the algorithm performs for inputs of size n.

For binary search, n is the number of candidates and a useful operation is one comparison with the middle value.
If two independent inputs grow, keep both variables: for example, n orders and m customers.
5 · Cases

Best, average, and worst case are different cost functions

Best case

Linear search finds the target first.

T_best(n) = 1

Average case

Requires assumptions about target position and presence.

T_avg(n) ≈ n / 2

Worst case

The target is last or absent.

T_worst(n) = n

Big-O itself does not mean “worst case.” State the case, then state a bound.

Ignore constant factors and lower-order terms for asymptotic shape

3n + 20

Θ(n)

5 · Rigorous notation

O, Ω, and Θ describe different kinds of asymptotic bounds

O(g(n))

Eventual upper bound: grows no faster than a constant multiple of g.

Ω(g(n))

Eventual lower bound: grows no slower than a constant multiple of g.

Θ(g(n))

Both bounds hold: a tight asymptotic classification.

T(n) = 3n + 7. For n ≥ 1: 3n ≤ T(n) ≤ 10n. Therefore T(n) ∈ Θ(n).
T(n) ∈ O(n²) is also true but loose. Big-O alone need not state the exact growth class.
5 · A lower bound

Comparison sorting needs enough decisions to distinguish n! orders

Decision-tree proof idea

  1. n distinct values can arrive in n! possible orders.
  2. One comparison has at most two outcomes.
  3. A binary decision tree with n! leaves needs height at least log₂(n!).
  4. log₂(n!) grows as Ω(n log n).
This bound applies to comparison sorts—not to counting or radix methods with additional assumptions.
5 · Correctness

A loop invariant explains why repeated steps produce the right answer

1 · Initialization

The invariant is true before the first iteration.

For max: first value is maximum of the one-item prefix.

2 · Maintenance

One iteration preserves the invariant.

Compare next value; update only if larger.

3 · Termination

When the loop ends, the invariant implies the goal.

The processed prefix is now the entire list.

Efficiency matters after correctness: a fast wrong answer is still wrong.
5 · Big-O is not a stopwatch

Same growth class can still mean different elapsed time

Pure Python loop

A clear one-item-at-a-time scan executed by the Python interpreter.

Θ(n)

Optimized library reduction

May perform the same asymptotic scan in optimized native code, depending on dtype and operation.

often Θ(n)

Implementation, hardware, cache behavior, and constants affect runtime. As inputs become large, better asymptotic growth usually dominates constant-factor improvements.

7 · Hash map example

Two Sum remembers the complement needed later

def two_sum(values, target):
    seen = {}

    for index, value in enumerate(values):
        needed = target - value
        if needed in seen:
            return seen[needed], index
        seen[value] = index

    return None

7 · Stack and queue

Queue · FIFO

First in, first out.

Breadth-first search, task scheduling, moving windows.

7 · Expression algorithm

A stack preserves precedence while converting infix to postfix

5 + 2 × 3 − 8
Infix5 + 2 * 3 - 8
→
Postfix5 2 3 * + 8 -

Postfix makes the evaluation order explicit without parentheses.

Shunting-yard rule

  1. Output numbers immediately.
  2. Push operators onto a stack.
  3. Before pushing a left-associative operator, pop operators with higher or equal precedence.
  4. Parentheses control when operators are popped.
A complete parser must also handle tokenization, decimals, unary minus, and right-associative exponentiation. Never use eval on untrusted input.
