1.1 · Checkpoint

What did you not write?

"Fetch at most five rows" — in Lesson 4 that would have been a loop with a counter and a break. Look at the SQL again. Where is the loop? Where is the counter?
There isn't one — and that is the point. You declared "at most five" and the engine chose how to satisfy it. In Lesson 4 you described the procedure step by step; in Lesson 2 pandas vectorized the steps but you still stitched them together; in SQL you state only the properties of the result. The engine picks the algorithm — and, as Lesson 7 hinted, it may even use an index to avoid scanning at all.
Declaring instead of describing is the whole personality difference between SQL and everything you have written so far.

1 · Checkpoint

Three ways to ask, one spectrum

Place the three styles you now know — a Lesson 4 loop, a Lesson 2 pandas one-liner, a SQL query — on a spectrum from "describes the procedure" to "declares the result". Where does pandas sit, and why?
Loop → pandas → SQL. The loop spells out every step. SQL states only the properties of the answer and lets the engine choose the steps. pandas lives in between: each operation is vectorized (you don't write the inner loop), but you still stitch the operations together in order — mask, then sort, then head. SQL hands even that sequencing to the engine, which is exactly why the next unit's evaluation-order surprises are possible.
Unit 1 gave you the sentence shape. Unit 2 adds the verbs that summarize: aggregates and GROUP BY.

2.1 · Whole-table aggregates

Five reductions in one SELECT

sql("""
    SELECT COUNT(*)                              AS count,
           ROUND(SUM(quantity * unit_price), 2)  AS sum,
           ROUND(AVG(unit_price), 2)             AS avg,
           MIN(unit_price)                       AS min,
           MAX(unit_price)                       AS max
    FROM transactions
""")
count      sum   avg   min    max
   60  1421.09  2.88  0.12  11.17

pandas twins: COUNT(*) is len(df), SUM(col) is df[col].sum(), AVG is .mean(), and so on down the list.

AS name

Renames the output column — the SQL twin of naming a result in agg.

ROUND(x, 2)

Is Python's round(x, 2); it just keeps the money readable.

Sixty rows in, one row out. Note that SUM(quantity * unit_price) aggregates a computed expression — revenue was never a stored column.
2.2 · GROUP BY, the idea

Split, apply, combine — Lesson 2's groupby by another name

SplitGROUP BY source_country folds the 60 rows into 3 piles — UK, EIRE, Netherlands
→
ApplySUM(quantity * unit_price) runs inside each pile, producing one number per pile
→
Combineone output row per group: (country, revenue)
You already own this idea. tx.groupby("source_country")["revenue"].sum() — Lesson 2, Unit 6 — is the same three steps. SQL just declares the split key in the query itself.
GROUP BY k + aggregate is groupby(k).agg(…). Same machine, new spelling.

2.2 · The grouping rule

Every SELECT column must be grouped or aggregated

In a GROUP BY source_country query, what should SELECT transaction_id return? The UK group contains 48 different transaction ids.
There is no single honest answer — one output row per group, but 48 candidate ids. That is why the rule exists: every column in SELECT must either be grouped by, or wrapped in an aggregate. A bare column that is neither has no one value per group, so the database refuses to guess. (This trips everyone exactly once; now it has tripped you on a slide instead of in production.)
Grouped or aggregated — no third option. The rule is the engine protecting you from an ambiguous answer.
2.3 · HAVING vs WHERE

Both filter. The difference is when

WHERE filters rows

It runs before grouping, on the raw rows. At WHERE time, no group exists yet.

HAVING filters groups

It runs after grouping, on the aggregated result — one verdict per pile.

Why you cannot put SUM(…) in WHERE

At WHERE time the groups do not exist, so there is nothing to sum yet. That is precisely why HAVING had to be invented. Interviewers love this one because getting it wrong is silent.

Interview answer in one line: WHERE filters rows, HAVING filters groups.
2.3 · WHERE first

Filter the raw rows, then group

sql("""
    SELECT source_country                        AS country,
           ROUND(SUM(quantity * unit_price), 2)  AS revenue
    FROM transactions
    WHERE unit_price >= 1.0    -- a condition on each RAW row
    GROUP BY source_country
    ORDER BY revenue DESC
""")
       country  revenue
United Kingdom  1220.07
          EIRE   148.88
   Netherlands    16.45

The cheap lines (price under 1.00) were dropped before any pile formed — so every country's total shrank a little (UK 1248.80 → 1220.07).

All three countries still appear — WHERE removed rows, not groups.
2.3 · HAVING after

Group everything first, then keep only the big piles

sql("""
    SELECT source_country                        AS country,
           ROUND(SUM(quantity * unit_price), 2)  AS revenue
    FROM transactions
    GROUP BY source_country
    HAVING SUM(quantity * unit_price) > 100   -- on each GROUP
    ORDER BY revenue DESC
""")
       country  revenue
United Kingdom  1248.80
          EIRE   148.88

Netherlands is gone. Its pile totalled 23.41 — it survived every row filter, formed a group, and was then dropped by HAVING > 100.

Same table, opposite timing: the aggregate condition could only be checked after the piles existed.
2.3 · The pandas twin makes the timing obvious

HAVING is simply the second mask

per_country = tx.groupby("source_country")["revenue"].sum()   # group first
per_country[per_country > 100]                                # HAVING = mask AFTER

Two masks, two moments

  • WHERE = the mask you apply to tx before groupby
  • HAVING = the mask you apply to the grouped result after
In pandas the timing was never confusing — the two masks are literally two separate lines. SQL packs both into one query, and only the keyword tells you which moment each filter runs at.
When a HAVING puzzle stumps you, translate to pandas: which line would the mask sit on?

3 · NULL and dates

Two topics that quietly decide whether your numbers are right

NULL

The database's word for "no value here" — the twin of pandas' NaN/NaT from Lesson 2. It behaves in comparisons like nothing you have met, and getting it wrong is silent.

Dates

SQLite stores dates as text; its date functions read that text. Slicing "which month?" out of a timestamp is what turns raw lines into a monthly report.

Both have Lesson 2 twins — isna/fillna, and dt.to_period("M"). The twins carry you most of the way; the one genuinely new thing is three-valued logic.
Nothing in this unit changes what a query looks like. It changes what your query silently misses.

3.1 · COUNT(*) vs COUNT(col)

One counts rows; the other skips the holes

sql("""
    SELECT COUNT(*)          AS all_rows,
           COUNT(country)    AS with_country,
           COUNT(quantity)   AS with_quantity,
           COUNT(order_date) AS with_date
    FROM orders_raw
""")
all_rows  with_country  with_quantity  with_date
       8             6              6          6

pandas twin — the exact same distinction: len(df) counts rows (8); df["country"].count() skips NaN (6). Same rule, same trap.

COUNT(*) asks "how many rows?"; COUNT(country) asks "how many rows have a country?" — it skips NULLs.
3.1 · Checkpoint

What does AVG divide by?

orders_raw has 8 rows; two quantities are NULL. AVG(quantity) — does it divide the total by 8, or by 6? And when is the answer it gives you wrong for your purposes?
By 6 — SUM and AVG skip NULLs just like COUNT(col) does, so AVG(quantity) = 26 ÷ 6 ≈ 4.33. That is correct if missing means "we don't know", but silently wrong if a missing quantity really means zero — then the true mean is 26 ÷ 8 = 3.25, and nothing errored to warn you. Whether a hole means "unknown" or "zero" is a data decision, not something the engine can know.
Aggregates skip NULLs silently. The number is only right if "skip" matches what missing means in your data.
3.2 · Three-valued logic

NULL is not a value — it is the absence of one

TRUE

The condition holds. WHERE keeps the row.

FALSE

The condition fails. The row is dropped.

UNKNOWN

The condition cannot be evaluated — some operand is NULL. The row is dropped too.

Why a third value exists

"Is an unknown country equal to 'EIRE'?" has no honest yes-or-no answer. Any comparison with NULL — =, <>, >, anything — returns UNKNOWN.

The rule that bites

WHERE keeps a row only when its condition is TRUE — never when it is UNKNOWN. So NULL rows fail both a test and its opposite.

Ordinary logic has two answers. Add NULL and you get a third — and WHERE treats it like FALSE.
3.2 · The demo

Two filters that look like exact opposites

uk    = sql("SELECT COUNT(*) AS n FROM orders_raw "
            "WHERE country =  'United Kingdom'")
notuk = sql("SELECT COUNT(*) AS n FROM orders_raw "
            "WHERE country <> 'United Kingdom'")
country =  'United Kingdom' : 4
country <> 'United Kingdom' : 2
rows in table                : 8
4 plus 2 is 6, not 8. The two rows with NULL country satisfy neither filter — for them, both conditions evaluate to UNKNOWN, and WHERE drops UNKNOWN rows.
This is the single most common NULL bug: a <> filter that silently loses the rows you never accounted for. Lesson 2 had the same moment — a NaN region matched no equality test.

3.3 · IS NULL

To find the holes, you need the special verb

-- Never  WHERE country = NULL  : that is UNKNOWN for every row.
sql("SELECT order_id, customer_id FROM orders_raw "
    "WHERE country IS NULL")
order_id customer_id
       3      C14646
       5      C13089

IS NULL is df["col"].isna(); IS NOT NULL is .notna(). They are the only tests that see a hole.

Why = NULL can never work

Equality with NULL evaluates to UNKNOWN — even NULL = NULL does. A hole is not a value you can compare; it is the absence of one. SQL therefore gives you a dedicated verb, IS NULL, that asks about absence itself.

Test for NULL with IS NULL / IS NOT NULL — never with = or <>.
3.3 · COALESCE

Fill the holes — on purpose, and honestly

-- COALESCE(x, fallback): the first non-NULL argument.
sql("""
    SELECT order_id,
           COALESCE(country, 'Unknown') AS country,
           COALESCE(quantity, 0)        AS quantity
    FROM orders_raw
    ORDER BY order_id
""")
order_id        country  quantity
       1 United Kingdom       6.0
       2           EIRE       0.0
       3        Unknown       4.0
       5        Unknown       0.0

Filling is a decision, not a reflex — Lesson 2's discipline still applies

COALESCE(quantity, 0) is only honest if a missing quantity truly means zero. Otherwise you have invented data — exactly the trap Lesson 2 warned about with missing prices.

COALESCE(col, fallback) is df["col"].fillna(fallback) — same power, same responsibility.

    SELECT strftime('%Y-%m', transaction_date)   AS month,

December 2010 dominates because that is where most of these pinned rows fall. pandas twin: tx.groupby("month")["revenue"].sum().

Notice what you just grouped by: a derived value (the month), not a stored column — the same move as grouping a pandas Series you computed on the fly. Add one more grouping key (country) and you have the exercise.

3 · Checkpoint

The audit question

A colleague reports: "I filtered WHERE region <> 'EU' and summed the revenue — non-EU revenue is 40,000." The region column has NULLs. What did they actually compute, and what two-line check exposes it?
They computed revenue over rows whose region is known and not EU — every NULL-region row was silently dropped, because NULL <> 'EU' is UNKNOWN. The check: SELECT COUNT(*) FROM t versus COUNT(region) — if the two differ, holes exist, and the honest report either adds OR region IS NULL, or fills with COALESCE(region, 'Unknown') and shows the Unknown bucket explicitly.

4 · Reading queries

A query is written top-to-bottom — but it does not run that way

How you write it

SELECT first, FROM second, WHERE third… the order that reads like English.

How the engine runs it

SELECT is nearly last. The rows must exist, be filtered, and be grouped before output columns can be computed.

Learning the real order turns SQL from spell-casting into something you can read and predict — and it explains, in one stroke, both "weird" errors beginners hit.
Unit 4's promise: by the end you will predict a query's output — or its error — before running it.
4 · The logical evaluation order

Six steps, always the same

1 · FROMpick the table (and joins) → the raw rows exist
2 · WHEREkeep rows matching a row condition → filter before grouping
3 · GROUP BYfold rows into groups → groups now exist
4 · HAVINGkeep groups matching a condition → filter after grouping
5 · SELECTcompute the output columns and aliases → aliases are born here
6 · ORDER BY / LIMITsort the finished rows, keep the first n
Read any query in this order — 1 through 6 — and its behavior stops being surprising.
4 · Two consequences

The order explains both things beginners find "weird"

An aggregate cannot appear in WHERE

WHERE (step 2) runs before GROUP BY (step 3), so SUM(…) has nothing to sum yet.

That is why HAVING (step 4) exists.

A SELECT alias cannot be trusted in WHERE

The alias is created at step 5; WHERE at step 2 has never heard of it.

But an alias is fine in ORDER BY (step 6) — by then it exists.

Neither rule needs memorizing. Both fall straight out of the six-step order: a clause can only see what earlier steps have already built.
One timeline, two "gotchas" dissolved. We now watch the timeline run, then prove both consequences in code.
4 · Animated trace

One query, clause by clause, in evaluation order

4 · Demonstration 1

An aggregate in WHERE is a genuine error

try:
    sql("""
        SELECT source_country, SUM(quantity * unit_price) AS revenue
        FROM transactions
        WHERE SUM(quantity * unit_price) > 100
        GROUP BY source_country
    """)
except Exception as error:
    print(type(error).__name__, "->", error)
DatabaseError
-> misuse of aggregate: SUM()

Diagnosis: WHERE runs before GROUP BY,
           so there is no SUM yet.
Fix:       move the aggregate condition
           into HAVING.
The engine refuses loudly — the good kind of failure (Lesson 5). The fix is never "work around it"; it is HAVING, the clause that runs at the right time.
4 · Demonstration 2

The alias trap that bites even SQLite

# The alias `unit_price` is the DOUBLED price. But WHERE
# filters on the ORIGINAL column, then SELECT doubles
# the survivors — not what the alias suggests.
sql("""
    SELECT transaction_id, unit_price * 2 AS unit_price
    FROM transactions
    WHERE unit_price > 5
    ORDER BY transaction_id
    LIMIT 4
""")
returned (doubled)     WHERE actually saw
   536381-04  13.5        536381-04  6.75
   536540-08  17.9        536540-08  8.95
   536540-09  17.9        536540-09  8.95
   536557-16  15.9        536557-16  7.95

SQLite is unusually lenient and lets aliases into WHERE; Postgres, SQL Server, and MySQL reject it. Worse: when the alias reuses a real column's name, WHERE silently reads the original column.

Every original price is above 5 — proof WHERE never touched the alias. Portable habit: never reference a SELECT alias in WHERE.
4 · Predict the output · 1 of 6

WHERE, then GROUP BY, then COUNT

SELECT source_country AS country, COUNT(*) AS n
FROM transactions
WHERE quantity >= 6
GROUP BY country
ORDER BY n DESC;
Read it in evaluation order and commit to a prediction: what is n counting?
WHERE quantity >= 6 keeps only the bulk lines first; then they are grouped and counted. So n is the number of bulk lines per country — United Kingdom 15, EIRE 4, Netherlands 2 — not all lines (48/6/6). WHERE shrank the rows before COUNT ever ran.
Write down your prediction before running a query. Committing first is what makes the surprise stick.
4 · Predict the output · 2 of 6

HAVING on a count

SELECT source_country AS country, COUNT(*) AS n
FROM transactions
GROUP BY country
HAVING COUNT(*) > 10;
No WHERE this time. How many rows come back, and which countries survive?
Every row is grouped (no WHERE); then HAVING COUNT(*) > 10 keeps only groups with more than ten lines. Only United Kingdom (48 lines) qualifies — EIRE and Netherlands have 6 each and are dropped. One row out. HAVING filtered the groups, after counting.
Q1 filtered rows before counting; Q2 filtered groups after. Same keywords, opposite moments.
4 · Predict the output · 3 of 6

Will this even run?

SELECT source_country, AVG(quantity) AS avg_q
FROM transactions
WHERE AVG(quantity) > 5
GROUP BY source_country;
Trick question or real query? Decide before peeking.
It errors: DatabaseError -> misuse of aggregate: AVG(). WHERE (step 2) runs before GROUP BY (step 3), so there is no average to compare against yet. The fix is HAVING AVG(quantity) > 5. This is Demonstration 1 again — now as a reflex.
If your condition mentions an aggregate, it belongs in HAVING. No exceptions.
4 · Predict the output · 4 of 6

An alias in ORDER BY

SELECT transaction_id,
       ROUND(quantity * unit_price, 2) AS line_total
FROM transactions
ORDER BY line_total DESC
LIMIT 3;
Q3's alias-in-WHERE failed. Is an alias in ORDER BY legal? What comes out, and in what order?
Legal — and it returns the three largest line totals, biggest first: 525.60, 208.80, 81.36. ORDER BY (step 6) runs after SELECT (step 5), so by then the alias exists. Contrast with Q3, where the alias-referencing condition sat in WHERE (step 2).
Aliases are born at step 5. Any clause that runs later may use them; any clause that runs earlier may not.
4 · Predict the output · 5 of 6

The alias reuses a column name

SELECT transaction_id, quantity * 10 AS quantity
FROM transactions
WHERE quantity >= 12
ORDER BY transaction_id
LIMIT 3;
The alias quantity shadows the real column. What does WHERE filter on, and what values come back?
WHERE filters on the original stored quantity (the alias does not exist at step 2), keeping rows with 12 or more units. SELECT then multiplies each survivor by 10 — so the returned values are 120, 240, 240: each is 10 × a stored value that was itself ≥ 12. The name collision is Demonstration 2's trap in miniature.
When an alias shadows a column, WHERE reads the column and never tells you. Avoid the collision entirely.
4 · Predict the output · 6 of 6

NULL meets a <> filter — the units combine

SELECT order_id, country
FROM orders_raw
WHERE country <> 'United Kingdom';
The orders_raw table again: 4 UK rows, 2 EIRE, 2 NULL. Which rows come back?
Only the two EIRE rows (order_id 2 and 8). The two NULL-country rows are not returned: NULL <> 'United Kingdom' is UNKNOWN, and WHERE keeps only TRUE. Unit 3's three-valued logic, resurfacing inside Unit 4's reading drill — exactly how it will resurface in real reports.
Six for six? Then you read queries the way the engine runs them. That skill is the whole unit.
Wrap-up · misconceptions

Five beliefs this lesson should have broken

MisconceptionReality
"SQL runs top to bottom, like a script"It runs FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY — SELECT is nearly last
"WHERE and HAVING are interchangeable"WHERE filters rows before grouping; HAVING filters groups after — only HAVING holds an aggregate
"COUNT(*) and COUNT(col) are the same"COUNT(*) counts rows; COUNT(col) skips NULL — just like len(df) vs df[col].count()
"x <> 'A' returns everything that isn't 'A'"Rows where x is NULL evaluate to UNKNOWN and are dropped by both = and <>
"An alias I wrote can be used anywhere"Aliases are born in SELECT; WHERE runs earlier and cannot see them (SQLite's leniency is a portability trap)
Each row of this table is a real bug someone ships weekly. You now know why all five happen.
