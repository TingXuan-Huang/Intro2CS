1 · INNER and LEFT JOIN

Two tables, linked by a key

transactions                      customers
+----------------+-----------+    +-------------+----------------+
| transaction_id | customer  |    | customer_id | country        |
| 536381-01      | C15311 ---+----+-> C15311    | United Kingdom |
+----------------+-----------+    +-------------+----------------+
              JOIN ... ON transactions.customer_id = customers.customer_id

Lesson 1's idea, again

customer_id lives in both tables: a transaction's key that points to a record — the dictionary lookup, now spanning two tables.

Lesson 7's name for it

In the customers table it is (in spirit) the primary key; in transactions it is a foreign key — a column whose job is to point into another table.

ON names the key both tables share. Everything a join does — keep, drop, multiply — follows from how that key matches.

1 · INNER JOIN

Keep only rows that match on both sides

customer_id  country

C13089       United Kingdom
...          ...

c and t are table aliases; JOIN alone means INNER JOIN. The question asked: which customers have at least one transaction?

Ten customers come back — but the table holds twelve. Something disappeared, and SQL did not say a word.
1 · Checkpoint

Where did the other two go?

Twelve customers in the table; ten in the result. What happened to C99001 and C99002 — and did SQL warn you?
An INNER JOIN keeps a row only when the key matches on both sides. C99001 and C99002 never placed an order, so no transaction row carries their id — and they silently vanish. No error, no warning, two rows gone. This is exactly what happened to the orphan sale in Lesson 2's merge: the default join quietly drops what it cannot match.
"Silently" is the important word. A dropped row looks identical to a row that never existed — unless you reconcile the counts.
1 · LEFT JOIN

Keep every row on the left, matched or not

run("""
SELECT c.customer_id, c.country,
       COUNT(t.transaction_id) AS n_transactions
FROM customers c
LEFT JOIN transactions t ON c.customer_id = t.customer_id
GROUP BY c.customer_id, c.country
ORDER BY n_transactions
""")

COUNT(t.transaction_id) counts non-NULL matches — so the no-order customers score 0 instead of disappearing.

C15311       United Kingdom  7

A LEFT JOIN keeps every left-hand row and fills the right-hand columns with NULL where nothing matched — the same hole merge filled with NaN in Lesson 2.

1 · Under the hood

You built this engine in Lesson 4 — it is a hash join

Buildput the smaller table's keys in a hash table (a dict): customer_id → row
→
Probescan the big table once; look each key up in the dict — O(1) per row
→
Emitoutput one joined row per match (or a NULL-filled row, for LEFT)

The naive way — O(n·m)

Compare every transaction against every customer: nested loops. 61 × 12 here; billions × millions in production.

The dict way — O(n+m)

One pass to build, one pass to probe. Lesson 4's hash-join exercise, verbatim — that is what the engine runs when you type JOIN.

Declarative means the engine picks the algorithm. You say JOIN ... ON; it chooses the O(n+m) plan you once wrote by hand.
1 · Pitfall — row multiplication

A join's output lives at the grain of the many side

run("""
SELECT COUNT(*) AS joined_rows
FROM customers c
JOIN transactions t ON c.customer_id = t.customer_id
""")
joined_rows
61

12 customers in, 61 rows out

Each customer row is repeated once per matching transaction. That is correct — and dangerous: sum a customer-level column after this join and you count it six or seven times.

Lesson 2 showed you the same fan-out when a merge key was not unique.

Before trusting any joined result: how many rows should it have? Reconcile the count — the Lesson 2 habit, now for joins.
Aggregate at the right grain: compute per-customer sums from transactions, then join the summary — not the other way round.
1 · Pitfall — NULL keys

NULL never equals anything — not even NULL

run("SELECT NULL = NULL AS are_they_equal")
are_they_equal
None   -- not 1, not 0: NULL

What that does to a join

A row whose join key is NULL matches nothing — ON a.key = b.key is never true for it. It silently drops out of an INNER JOIN, exactly like an unmatched key.

Our fixture ids are all present; real exports are not so kind.

The check

WHERE key IS NULL on each table before joining — count the rows that cannot possibly match.

The Lesson 7 fix

Declare the key NOT NULL in the schema, and the database refuses such rows at the door.

Two ways a join loses rows silently: keys that match nothing, and keys that are NULL. Both are found the same way — reconcile the counts.

1 · Checkpoint

Write the shape from memory

An interviewer asks: "We have a products table and a sales table. Find the products that have never sold." What is the query shape — and why would an INNER JOIN be exactly wrong?
SELECT p.* FROM products p LEFT JOIN sales s ON p.product_id = s.product_id WHERE s.sale_id IS NULL. The LEFT JOIN keeps every product; the IS NULL keeps only the ones with no matching sale. An INNER JOIN would be exactly wrong because it keeps only matched rows — the never-sold products are the rows it deletes, so the very answer you want is the part it throws away.
Unit 1 gave you the row-matching verbs. Unit 2 adds the per-row conditional — SQL's if/else.
2 · CASE WHEN

SQL's if/else, applied to every row

run("""
SELECT transaction_id, quantity,
       CASE WHEN quantity >= 3 THEN 'bulk'
            ELSE 'single' END AS order_size
FROM transactions
LIMIT 6
""")
transaction_id  quantity  order_size
536381-01       1         single
536381-02       3         bulk
536381-03       2         single
536381-04       10        bulk
536381-05       1         single
536381-06       2         single

The declarative twin of…

Lesson 2's df.loc[cond, col] = ... and NumPy's np.where — one expression labels every row, no loop written.

Anatomy

CASE WHEN cond THEN a ELSE b END — conditions checked top to bottom, first match wins, END closes it.

A CASE WHEN is a new computed column, written inline in the SELECT — the "conditional column" you built in Lesson 2.
2 · Conditional aggregation

Put the CASE WHEN inside the SUM

run("""
SELECT source_country,
       SUM(CASE WHEN quantity >= 12 THEN 1 ELSE 0 END) AS bulk_lines,
       SUM(CASE WHEN quantity <  12 THEN 1 ELSE 0 END) AS small_lines,
       COUNT(*) AS total_lines
FROM transactions
GROUP BY source_country
""")
source_country  bulk_lines  small_lines  total_lines
EIRE            2           4            6
Netherlands     2           4            6
United Kingdom  13          36           49

Each row contributes 1 or 0 to each sum — so every SUM(CASE WHEN ...) is a count of rows meeting a condition, per group.

The trick that surprises people: one GROUP BY, several conditional sums — and you have built a little cross-tab in a single pass.
2 · Checkpoint

Where have you built this table before?

Countries down the side, "bulk vs small" across the top, counts in the cells. Which two earlier tools built exactly this shape — and what did Lesson 4 call the underlying scan?
Lesson 2's pivot_table / crosstab built this shape from a DataFrame. A pivot table is not a special object — it is conditional counting per group, and SQL says so explicitly.
2 · Segmentation

Bucket each customer by how much they spent

run("""
SELECT customer_id,
       ROUND(SUM(quantity * unit_price), 2) AS total,

            WHEN SUM(quantity * unit_price) >= 100 THEN 'mid'
            ELSE 'low' END AS band
FROM transactions

ORDER BY total DESC
""")

C15311       91.52   low
C12748       24.75   low
...          ...     low

A CASE WHEN can wrap an aggregate: the branches test each group's SUM, not each row.

Multiple WHEN branches are checked in order — put the highest threshold first, and ELSE catches the rest.

2 · One honesty note

The duplicate is still in every total

What it is doing right now

The duplicated line adds £1.95 to customer C15311's total. Every SUM so far — bands, cross-tabs, totals — is very slightly wrong.

Why we have not panicked

£1.95 changes no band here — but only because we checked. On real data you will not be so lucky, and you will not know unless you reconcile.

2 · Checkpoint

Unit 2 in one breath

A colleague asks: "How many transactions per country were under £5 of revenue, and how many £5 or over — one row per country?" Say the query shape out loud before revealing.
SELECT source_country, SUM(CASE WHEN quantity * unit_price < 5 THEN 1 ELSE 0 END) AS under_5, SUM(CASE WHEN quantity * unit_price >= 5 THEN 1 ELSE 0 END) AS over_5 FROM transactions GROUP BY source_country — one GROUP BY, two conditional sums, a cross-tab in a single pass. 3 · CTEs

WITH name AS (…) — give the intermediate result a name

run("""
WITH customer_totals AS (
    SELECT customer_id, SUM(quantity * unit_price) AS total
    FROM transactions
    GROUP BY customer_id
)
SELECT CASE WHEN total >= 300 THEN 'high'
            WHEN total >= 100 THEN 'mid'
            ELSE 'low' END AS band,
       COUNT(*) AS n_customers
FROM customer_totals
GROUP BY band
""")
band  n_customers
high  1
low   7
mid   2
-- identical to the nested version

A CTE — Common Table Expression — is Unit 2's inner query, lifted out and named. The outer query reads it like any table.

Now it reads top-to-bottom: "first compute customer_totals, then band them and count." Same answer, honest order.
3 · The Lesson 5 move

A subquery is an anonymous lambda; a CTE is a named function

Inside-out · the nested subquery
SELECT band, COUNT(*)
FROM (
    SELECT ... FROM transactions ...   ← no name,
)                                        read me first
GROUP BY band
Top-to-bottom · the CTE
WITH customer_totals AS ( ... )   ← named, first
SELECT band, COUNT(*)
FROM customer_totals              ← then used
GROUP BY band

Exactly Lesson 5's refactor

You extracted a well-named function so a long computation read top-to-bottom. WITH is the same move for queries.

The pandas twin

A CTE is a named intermediate DataFrame: customer_totals = df.groupby(...)..., then work with customer_totals.

Readability is a correctness tool: a query you can read top-to-bottom is a query you can check.
3 · Named once, used twice

Customers who spent above the average customer

run("""
WITH customer_totals AS (
    SELECT customer_id, SUM(quantity * unit_price) AS total
    FROM transactions
    GROUP BY customer_id
)
SELECT customer_id, ROUND(total, 2) AS total
FROM customer_totals
WHERE total > (SELECT AVG(total) FROM customer_totals)
ORDER BY total DESC
""")
customer_id  total
C13089       868.26
C14298       181.20
C14911       148.88
-- average customer total: 142.30

customer_totals is defined once and referenced twice — in the FROM and inside the WHERE's subquery.

Without the CTE you would write the same aggregation out twice — and any change would have to be made in both places. Don't Repeat Yourself, in SQL.

3 · Checkpoint

When do you reach for a CTE?

Two separate reasons came up in this unit for replacing a nested subquery with WITH name AS (…). What are they — and which Lesson 5 principle does each one echo?
Readability: the logic reads top-to-bottom — "first compute this, then use it" — instead of inside-out; that is Lesson 5's extract a well-named function. Reuse: a result defined once can be referenced twice (the FROM and the AVG), so a change happens in one place; that is Lesson 5's Don't Repeat Yourself. Same two arguments, same order, new language.
CTEs also stack — WITH a AS (...), b AS (...) — which is exactly what makes the next unit's queries readable.

4 · The window

A window computes across the group but keeps every row

SELECT transaction_id, customer_id,

FROM transactions
ORDER BY customer_id
LIMIT 8
""")

4 · The trap

The #1 SQL interview mistake, stated once

"One number per group" — or — "for each row, a number computed across its group." Which question calls for GROUP BY, and which for a window?
GROUP BY answers "one number per group" and loses the rows. A window (... OVER (PARTITION BY key)) answers "for each row, a number computed across its group" and keeps the rows. PARTITION BY is GROUP BY's window twin. Interviewers love the difference because reaching for GROUP BY when you needed a window — or vice-versa — is the single most common SQL mistake.

"Total revenue per customer"

One row per customer → GROUP BY.

"Each line's share of its customer's total"

Needs the line and the total → window.

Ask of every question: does the answer have one row per group, or one row per original row? That decides the tool.

4 · Top-N per group

Impossible for plain GROUP BY, trivial for a window

run("""
WITH per AS (
    SELECT t.customer_id, c.country,
           SUM(t.quantity * t.unit_price) AS total
    FROM transactions t
    JOIN customers c ON t.customer_id = c.customer_id
    GROUP BY t.customer_id, c.country
),
ranked AS (
    SELECT country, customer_id, total,
           ROW_NUMBER() OVER (PARTITION BY country
                              ORDER BY total DESC) AS rn
    FROM per
)
SELECT country, customer_id, ROUND(total, 2) AS total
FROM ranked
WHERE rn <= 3
ORDER BY country, rn
""")
country         customer_id  total
EIRE            C14911       148.88
Netherlands     C14646       23.41
United Kingdom  C13089       868.26
United Kingdom  C14298       181.20
United Kingdom  C15311       91.52

CTEs stack, as promised: per aggregates, ranked numbers, the final SELECT filters rn <= 3.

"Top 3 per country" cannot be a bare GROUP BY — it would collapse the customers away. Rank with a window, then filter the rank: the canonical answer.
4 · Ties

ROW_NUMBER vs RANK vs DENSE_RANK

FunctionOn a tie for 1st…SequenceUse when
ROW_NUMBER()breaks it arbitrarily1, 2, 3, 4you need exactly N rows (top-N, de-dup)
RANK()both get 1; next rank skips1, 1, 3, 4ties share a place, gaps show how many tied
DENSE_RANK()both get 1; no gap1, 1, 2, 3"Nth highest value" — ties share a place, no gaps
Same window syntax for all three — OVER (PARTITION BY ... ORDER BY ...). The only difference is what happens on a tie. Pick by how you want ties treated.
Unit 5's "2nd highest" drill uses DENSE_RANK for exactly this reason: two customers tied at the top should both count as 1st.

The loose thread: find it, number it, keep copy 1

Step 1 · find the duplicate
run("""
SELECT transaction_id, COUNT(*) AS copies
FROM transactions
GROUP BY transaction_id
HAVING COUNT(*) > 1
""")
transaction_id  copies
536381-01       2
Step 2 · number the copies, keep the first
run("""
WITH numbered AS (
    SELECT transaction_id, customer_id, quantity, unit_price,
           ROW_NUMBER() OVER (PARTITION BY transaction_id
                              ORDER BY rowid) AS copy_no
    FROM transactions
)
SELECT COUNT(*) AS rows_kept
FROM numbered
WHERE copy_no = 1
""")
rows_kept
60
PARTITION BY transaction_id makes each id its own little group; ROW_NUMBER counts its copies 1, 2, …; keeping copy_no = 1 keeps one row per id.

4 · Reconcile the total

What the duplicate was quietly costing us

run("""
SELECT
    ROUND((SELECT SUM(quantity * unit_price)
           FROM transactions), 2) AS raw_total,
    ROUND((SELECT SUM(quantity * unit_price)
           FROM (SELECT DISTINCT * FROM transactions)), 2)
      AS deduped_total
""")
raw_total  deduped_total
1423.04    1421.09
-- difference: exactly £1.95

The difference is the one duplicated line, double-counted — the £1.95 we predicted in Unit 2's honesty note.

Uniqueness is a property (Lesson 1). The duplicate silently violated it and inflated revenue until ROW_NUMBER() = 1 restored it. A real PRIMARY KEY would have refused the row at the door (Lesson 7).
4 · Checkpoint

Which tool, and why?

Your manager asks for "each transaction line, with the percentage it contributes to its customer's total spend." GROUP BY or window — and roughly what does the query look like?
A window: the answer has one row per transaction line, but needs a number computed across the customer's group — both at once. Shape: SELECT transaction_id, quantity * unit_price / SUM(quantity * unit_price) OVER (PARTITION BY customer_id) AS share FROM transactions. A GROUP BY would collapse the lines away and leave nothing to attach the percentage to.
Unit 4 gave you partitions, ranking, running totals, and de-dup. Unit 5 dresses them up as the four questions interviewers actually ask.

5 · Drill 1 — Nth highest

"Find the 2nd highest spender"

run("""
WITH per AS (
    SELECT customer_id, SUM(quantity * unit_price) AS total
    FROM (SELECT DISTINCT * FROM transactions)
    GROUP BY customer_id
),
ranked AS (SELECT *, DENSE_RANK() OVER (ORDER BY total DESC) AS rnk
           FROM per)
SELECT customer_id, ROUND(total, 2) AS total
FROM ranked WHERE rnk = 2
""")
customer_id  total
C14298       181.2

Note the hygiene: SELECT DISTINCT * feeds the drill de-duplicated data — Unit 4's cleanup, now a habit.

Swap the 2 for any N. The classic "2nd highest salary" is this exact query with a different column name. DENSE_RANK so ties share a place.

HAVING filters groups after aggregation — Lesson 8's distinction from WHERE, which filters rows before it.

5 · Drill 3 — period over period

"Month-over-month change" — LAG reaches back one row

run("""
WITH monthly AS (
    SELECT substr(transaction_date, 1, 7) AS month,
           SUM(quantity * unit_price) AS revenue
    FROM (SELECT DISTINCT * FROM transactions)
    GROUP BY month
)
SELECT month, ROUND(revenue, 2) AS revenue,
       ROUND(revenue - LAG(revenue) OVER (ORDER BY month), 2)
         AS mom_change
FROM monthly
ORDER BY month
""")
month    revenue  mom_change
2010-12  1382.19  None
2011-01  17.06    -1365.13
2011-08  21.84    4.78

The first month has no previous row, so LAG returns NULL and the change is NULL — the honest answer, not a fake zero.

LAG(x) OVER (ORDER BY t) is the previous row's value — pandas' shift(). LEAD is the same move looking forward.

5 · Drill 4 — event gaps

"Longest gap between a customer's purchases"

run("""
WITH dates AS (SELECT DISTINCT customer_id, transaction_date
               FROM transactions),
gaps AS (
    SELECT customer_id,
           julianday(transaction_date)
             - julianday(LAG(transaction_date)
                 OVER (PARTITION BY customer_id
                       ORDER BY transaction_date)) AS gap_days
    FROM dates
)
SELECT customer_id, MAX(gap_days) AS longest_gap_days
FROM gaps
WHERE gap_days IS NOT NULL
GROUP BY customer_id
ORDER BY longest_gap_days DESC
""")
customer_id  longest_gap_days
C12748       3.0
C13089       1.0

Only two customers bought on more than one day, so only they have a gap — everyone else's LAG is NULL and drops out.

LAG per customer (PARTITION BY), difference the dates (julianday), aggregate the gaps. The same pattern powers "days between logins," "time to repeat purchase," and churn windows.

The lesson, one sentence

A relational query is merge and groupby written declaratively: say which rows to keep, how to bucket them, name the pieces — and compute across a group without losing the rows.

JOIN · CASE WHEN · WITH · window functions. Four clauses, one habit: reconcile the counts, because joins change the row set silently.
